Microservice Fund Management — la gestion des demandes de fonds d'une agence bancaire, expliquée de A à Z

Ce document part de zéro : il explique d'abord le métier (pourquoi une agence bancaire demande de l'argent liquide, avec quels mots), puis la technique (comment le code est organisé), puis comment installer, tester et améliorer le service. Chaque terme métier est défini au moment où il apparaît.

01 · Métier

Le métier, expliqué de zéro

Imaginez une agence bancaire un lundi matin. Trois situations classiques :

  • Le GAB (le distributeur automatique de billets, en anglais ATM) est presque vide : il faut le réapprovisionner en billets.
  • La caisse du guichet (là où le caissier sert les clients) n'a plus assez de billets de 200 DH : elle aussi doit être alimentée.
  • À l'inverse, après un gros dépôt d'un commerçant, la caisse contient trop d'argent. Une agence n'a pas le droit de garder des sommes illimitées (règle de sécurité) : il faut renvoyer l'excédent vers le centre de traitement des espèces de la banque.

Dans les deux premiers cas, l'agence demande qu'on lui apporte de l'argent : c'est une alimentation (dans le code : SUPPLY). Dans le troisième, elle demande qu'on vienne chercher de l'argent : c'est un délestage ou ramassage (dans le code : COLLECTION).

Définition — Demande de fonds Une demande de fonds (CashRequest) est le formulaire officiel par lequel une agence demande un mouvement d'argent liquide. Elle est traçable (qui a demandé, quand, pour quelle agence) car on parle d'argent physique : tout doit être auditable.

Point essentiel : on ne demande jamais « 250 000 dirhams » tout court. L'argent liquide est physique — le centre de traitement doit savoir exactement quoi mettre dans la valise : combien de billets de 200, combien de billets de 100, combien de pièces de 10. Ce détail s'appelle la ventilation par coupures.

Définition — Coupure Une coupure (Denomination) est un type précis de billet ou de pièce : « le billet de 200 DH », « la pièce de 10 DH ». Chaque coupure a une valeur faciale (faceValue : le montant écrit dessus, ex. 200,00) et un type : billet (BANKNOTE) ou pièce (COIN).

Enfin, une agence peut manipuler plusieurs devises (dirham MAD, euro EUR, dollar USD…), par exemple pour le change au guichet. Une même demande peut donc mélanger plusieurs devises, chacune avec sa propre ventilation.

À retenir Tout le microservice tourne autour de cet arbre. Créer une demande = enregistrer cet arbre entier. Annuler une demande = éteindre toutes ses branches. Le total déclaré doit toujours être cohérent avec la somme (valeur faciale × quantité) de chaque ligne.
02 · Métier

Vocabulaire métier ⇄ code

Le dictionnaire à garder sous la main. Colonne de gauche : le mot que dira un banquier. Colonne de droite : où il vit dans le code.

Demande de fonds — CashRequest
Le formulaire global. Porte : le code de l'agence demandeuse (branchCode), l'établissement, l'utilisateur qui a fait la demande (uidUser) et un statut global. C'est l'agrégat racine : la porte d'entrée unique pour tout modifier (voir §5).
Opération — CashOperation
Un mouvement d'argent précis à l'intérieur de la demande. Porte deux informations clés : le sens (operationSens) et la cible (operationType), plus un drapeau d'urgence (isUrgent : YES/NO), un numéro de caisse source (cashDeskNumber) et éventuellement une caisse cible (targetCashDeskNumber, pour un transfert de caisse à caisse).
Sens de l'opération — RequestType { SUPPLY, COLLECTION }
SUPPLY = alimentation : « apportez-moi de l'argent ». COLLECTION = délestage/ramassage : « venez chercher de l'argent ». (Le nom de l'enum est trompeur — il décrit le sens d'une opération, pas le type de la demande. Candidat au renommage, voir §12.)
Cible de l'opération — OperationType { ATM, CASH_DESK, CURRENCY }
Ce qu'on alimente ou déleste : ATM = le GAB (distributeur), CASH_DESK = la caisse du guichet, CURRENCY = la caisse de change (devises étrangères).
Détail devise — OperationCurrencyDetail
La part d'une opération exprimée dans une devise, avec son sous-total (totalAmountByCurrency). Permet à une même opération de mélanger MAD + EUR + USD.
Ligne de ventilation — OperationDenomination
La ligne « telle coupure × telle quantité ». Exemple : Billet 200 DH × 1000. C'est le niveau le plus fin, celui qui permet physiquement de préparer la valise.
Coupure — Denomination (référentiel)
Donnée de référence : type (billet/pièce), valeur faciale, libellé (« Billet 200 DH »). Référentiel = table de référence pré-remplie, qu'on consulte mais qu'on ne crée pas via l'API.
Devise — Currency (référentiel)
Code ISO à 3 lettres (MAD, EUR, USD) + nom. Le code envoie « MAD » et le service retrouve la devise en base.
Établissement — EstablishmentEntity (référentiel)
L'entité bancaire concernée (code à 3 caractères + nom). L'API reçoit le code (« 001 ») et le service le résout en base.
Profil utilisateur — UserProfileEntity (référentiel)
L'utilisateur demandeur, identifié par son uid, avec un profil (code, libellé, type) — probablement la brique des futures habilitations (qui a le droit de faire quoi).
03 · Métier

Cycle de vie et statuts

Une demande (et chacune de ses opérations) porte un statut qui raconte où elle en est dans sa vie :

  • INITIATED (« initiée ») : la demande vient d'être créée par l'agence, elle attend d'être traitée.
  • VALIDATED (« validée ») : la demande a été approuvée — par exemple par le responsable d'agence ou le centre des espèces. Aujourd'hui aucun endpoint ne fait cette transition ; elle est prévue mais pas encore implémentée (voir §13 et l'intégration Camunda).
  • CANCELLED (« annulée ») : la demande est abandonnée. Une demande annulée est gelée : plus aucune modification possible.

04 · Métier

Les règles métier, une par une

Chaque règle est expliquée avec son pourquoi métier et l'endroit du code qui la porte.

N°RèglePourquoi (métier)Où (code)
R1Une demande doit contenir au moins une opération.Une demande vide n'a aucun sens : on demande forcément quelque chose.CashRequest.validateRequest() (⚠️ écrite mais l'appel est commenté dans le flux de création — à réactiver)
R2Une demande CANCELLED est gelée : ni ajout d'opération, ni modification, ni update global.Une fois annulée, la demande ne doit plus bouger — sinon la piste d'audit devient incohérente.CashRequest.addOperation(), updateOperation() + garde dans CashDemandUseCaseImpl.update()
R3Annuler une demande annule toutes ses opérations (cascade).Le centre des espèces ne doit pas préparer une valise pour une opération dont la demande mère est morte.CashDemandRepositoryImpl.cancelById()
R4On peut annuler des opérations individuellement ; quand la dernière opération active est annulée, la demande bascule automatiquement en CANCELLED.L'agence peut renoncer à une partie de sa demande sans tout jeter. Mais une demande sans opération active est de facto morte.CashDemandRepositoryImpl.cancelOperations()
R5Une opération déjà annulée ne peut pas être annulée deux fois (erreur explicite).Détecter les doubles clics / doubles appels et garder un historique propre.cancelOperations() — lève "operation already cancelled"
R6Pour une alimentation de caisse (SUPPLY + CASH_DESK), la ventilation par coupures est obligatoire et le total doit se recalculer : Σ (valeur faciale × quantité).C'est la règle d'or du métier : on ne transporte pas « un montant », on transporte des billets comptés. Le total déclaré doit être prouvable ligne par ligne.CashOperation.validateAndCalculateTotal() (⚠️ actuellement buguée — voir I-02 en §12)
R7Les données de référence sont résolues, jamais inventées : devise par code ISO, coupure par libellé, établissement par code, utilisateur par uid. Inconnu ⇒ erreur immédiate.Empêche de créer une demande pour un « Billet 300 DH » qui n'existe pas ou une devise fantaisiste.CashRequestDataAccessMappers.toPersistence() — lève "la devise X n'existe pas" / "le billet Y n'existe pas"
Définition — Référentiel Un référentiel est une table de données de référence, alimentée par l'administration (pas par l'API) : la liste des devises, des coupures, des établissements, des utilisateurs. Les demandes pointent vers ces référentiels au lieu de recopier leurs valeurs — c'est ce qui garantit la cohérence (règle R7).
05 · Technique

Architecture — hexagonale (ports & adaptateurs)

Le projet suit l'architecture hexagonale (aussi appelée Clean Architecture ou ports & adaptateurs). L'idée en une phrase : le cœur métier ne doit dépendre de rien de technique. Ni de Spring, ni de la base de données, ni du web. On doit pouvoir changer PostgreSQL pour Oracle, ou REST pour Kafka, sans toucher une ligne des règles métier.

Définition — Port et adaptateur Un port est une interface Java déclarée par le cœur : « voilà ce dont j'ai besoin » (ex. CashDemandRepository : « sache sauvegarder une demande ») ou « voilà ce que je sais faire » (ex. CashDemandUseCase : « je sais créer/annuler une demande »). Un adaptateur est la pièce technique qui se branche sur un port : le contrôleur REST est l'adaptateur entrant (il pilote le cœur), l'implémentation JPA est l'adaptateur sortant (elle est pilotée par le cœur).

06 · Technique

Les 5 modules Maven

Le projet est un multi-module Maven : POM racine microservice-fund-management (hérite de spring-boot-starter-parent 3.3.1, Java 17). Chaque module = une couche de l'hexagone :

Module
Rôle
domain-core
Le cœur. Modèle métier pur : agrégat CashRequest, entités, enums, identifiants typés (CashRequestId…), règles R1→R7. Ne devrait dépendre de rien (seul Lombok est déclaré — mais des imports jakarta.persistence ont fui dedans, voir I-08).
application-service
L'orchestration. Cas d'usage (CashDemandUseCaseImpl : créer, lire, modifier, annuler), DTOs, mappers web, et les ports sortants (interfaces CashDemandRepository, DenominationRepository) que la couche data devra implémenter.
data-access
L'adaptateur base de données. Entités JPA (toutes héritent de BaseEntity : id UUID généré + dates de création/modification automatiques), repositories Spring Data, mappers domaine⇄entité, et les implémentations des ports.
application
L'adaptateur REST. Le contrôleur CashDemandController sur /cash (media type versionné application/vnd.api.v1+json) + son test MockMvc.
container
L'assembleur. Classe main WorkFlowFundManagementApplication (avec @EntityScan / @EnableJpaRepositories pointés sur data-access), le fichier application.yml, le plugin spring-boot-maven — c'est le seul module qui produit un jar exécutable.
Détail d'arborescence Le POM racine référence un module intermédiaire fund-management-domain, parent de domain-core et application-service. L'arbre réel : racine → {application, container, data-access, domain → {domain-core, application-service}}.

4 tables « transactionnelles » (en haut à gauche, l'arbre de la demande) + 4 tables « référentiels » (devise, coupure, établissement, utilisateur). Cascade ALL + orphanRemoval : sauver/supprimer la demande propage sur tout l'arbre.
Dérive de schéma : le script SQL du repo ne correspond plus aux entités Le script DDL présent dans le projet décrit un ancien modèle (colonnes unique_request_ref, requesting_branch, devise directement sur l'opération, pivot N:N à clé composée…). Les entités ont évolué depuis. Comme ddl-auto: update ajoute mais ne supprime jamais, une base initialisée avec le vieux script puis « complétée » par Hibernate contient un mélange de colonnes vivantes et mortes — et les colonnes mortes en NOT NULL font échouer les insertions. ⇒ C'est exactement le point à vérifier avec le script que ton tech lead va te fournir : s'il contient unique_request_ref, il est obsolète. Solution durable : Flyway (§14).

Stratégie de chargement

  • Toutes les relations sont LAZY (chargées à la demande) et open-in-view: false — bien : on est obligé de choisir consciemment ce qu'on charge.
  • La requête findByIdWithOperation fait un join fetch sur les opérations, mais pas sur les détails devise/coupures ⇒ risque de LazyInitializationException (erreur classique : « la session est fermée, je ne peux plus charger ») si on mappe la profondeur hors transaction.
  • À la création/modification, le service précharge tout le référentiel coupures + devises en mémoire (des Map) pour résoudre les libellés/codes ISO — acceptable pour de petits référentiels.
08 · Technique

L'API REST — CashDemandController

Chemin de base /cash. Tous les endpoints produisent le media type versionné application/vnd.api.v1+json — pense à envoyer un en-tête Accept compatible, sinon Spring répond 406 Not Acceptable.

EndpointCe qu'il fait, en françaisÉtat

PUT /cash/{id}Remplacer une demande : les opérations existantes sont supprimées et remplacées par celles du corps de la requête. Refusé si la demande est CANCELLED (R2).Fonctionne*

Installation & lancement

Prérequis

  • JDK 17 · Maven 3.8+ · PostgreSQL 13+ (local, via pgAdmin comme convenu avec ton collègue)
  • IDE avec le traitement d'annotations Lombok activé (tout le code en dépend)

1 · Créer la base dans pgAdmin

  1. pgAdmin → serveur local (localhost:5432) → clic droit Databases → Create → Database → nom : qlick_flow.
  2. Clic droit sur qlick_flow → Query Tool → coller/ouvrir le script du tech lead → Exécuter (F5).
  3. Vérifier sous Schemas → public → Tables que les tables existent, et surtout que les référentiels sont remplis (currency, denomination, establishment, user_profile) — sans eux, la création échoue (règle R7).
Vérifier le script reçu Si le script contient les colonnes unique_request_ref / requesting_branch sur cash_request, c'est l'ancienne version (voir §7) : signale-le, sinon les INSERT échoueront sur des NOT NULL fantômes.

2 · Configurer la datasource en local

Dans fund-management-container/src/main/resources/application.yml (aujourd'hui pointé sur une IP distante 10.74.234.132) :

spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/qlick_flow?binaryTransfer=true&reWriteBatchedInserts=true
    username: postgres
    password: <ton mot de passe pgAdmin>

⚠️ Ne pas committer ce changement — c'est ta config locale. (Cible long terme : variables d'environnement + profils Spring, voir §12/I-06.)

3 · Compiler et lancer

# depuis la racine du repo
mvn clean install -DskipTests        # installe les 5 modules dans le repo local Maven

cd fund-management-container         # seul module exécutable
mvn spring-boot:run
# ou : java -jar target/fund-management-container.jar

Le service écoute sur http://localhost:8300. show-sql: true affiche chaque requête SQL en console — pratique pour apprendre, à désactiver en production.

Frictions de build connues Le POM racine référence ${maven-compiler-plugin.version} sans définir la propriété (à épingler, ex. 3.13.0). Et ResourcePersistenceJpaRepositoryTest est un @SpringBootTest qui démarre tout le contexte, donc exige la base — d'où le -DskipTests tant qu'il n'est pas corrigé (Testcontainers, §14).

Problèmes actuels — l'audit

Classés par gravité. Bloquant = casse des requêtes dès aujourd'hui · Majeur = comportement ou qualité de données corrompus · Mineur = hygiène de code.

IDGravitéProblème & correctif
I-01BloquantgetAll() déclare un @PathVariable UUID id sur GET /cash, route qui n'a pas de variable de chemin ⇒ chaque appel de la liste échoue en 500. Correctif : supprimer le paramètre. Et trancher entre GET /cash/{id} et GET /cash/getCashDemandById — garder le RESTful, supprimer le doublon.
I-02BloquantLa règle R6 est buguée : faceValuesMap.get(opDenom.getDenomination()) cherche un objet Denomination comme clé dans une Map<String, BigDecimal> indexée par libellé ⇒ toujours null ⇒ toute création SUPPLY/CASH_DESK avec ventilation lève « cupure inconu ». Correctif : .getDenominationLabel() comme clé, stocker le total calculé, le comparer au total déclaré, et faire utiliser this à la méthode au lieu de se recevoir elle-même en paramètre.
I-03BloquantDérive de schéma + ddl-auto: update (§7) : les colonnes NOT NULL héritées du vieux script (unique_request_ref…) font échouer les insertions alors que le code est correct. Correctif : migration Flyway de référence générée depuis les entités actuelles, puis ddl-auto: validate.
I-04MajeurLes lectures passent par l'ancien mapper (CashRequestManualMapper) : les GET omettent détails devise, utilisateur, établissement. Et getAll() mappe la collection lazy operations hors transaction (open-in-view=false) ⇒ risque de LazyInitializationException. Correctif : tout router sur la nouvelle génération, ajouter les fetch-joins nécessaires, supprimer les anciens mappers.
I-05MajeurPas de contrat d'erreur : tout échec métier devient SessionServiceException (nom hérité d'un autre service !) ou RuntimeException ⇒ HTTP 500 pour « introuvable », « déjà annulée », « demande gelée ». Correctif : exceptions dédiées (CashRequestNotFoundException, OperationAlreadyCancelledException…) + @RestControllerAdvice qui les traduit en 404 / 409 / 422 avec un corps ProblemDetail (intégré à Spring 6).
I-06MajeurIdentifiants + IP interne en dur dans application.yml, commités dans le repo. Correctif : placeholders d'environnement (${DB_URL}…), profils Spring (local/dev/prod), secrets hors git. Désactiver show-sql hors dev.
I-07MajeurAucune validation d'entrée : starter-validation est dans le classpath mais zéro @Valid/contrainte. Un create sans établissement/utilisateur meurt en base sur une FK. Et update() fait getRequestStatus().toString() ⇒ NPE si le statut est absent. Correctif : @NotBlank/@NotEmpty/@Valid en cascade sur l'arbre DTO + gardes null-safe.
I-08MajeurPureté du domaine cassée : des classes de domain-core importent jakarta.persistence (CashRequest, Currency, Denomination…) — ça compile uniquement par fuite transitive. Correctif : purger ces imports + règle maven-enforcer (dépendances bannies) pour verrouiller.
I-09MajeurStratégie d'update risquée : clear() de la collection puis ré-ajout d'entités qui gardent leurs anciens ids ⇒ Hibernate mélange suppression d'orphelins et ré-insertion des mêmes clés (erreurs « identifier altered » / contraintes) dès que le front renverra les ids. Correctif : fusion par id (modifier l'existant, ajouter le nouveau, retirer l'absent) — l'ébauche du vieux CashOperationDataAccessMapper.updateEntityFromDomain avait la bonne idée, à finaliser sur le nouveau modèle.
I-10MineurCode mort / dupliqué : deux générations de mappers (le paquet mapperTemp1 ne doit pas partir en prod avec ce nom), CashDemandDTO inutilisé (et rangé dans dto.enums !), enums copiés du service identité (IdentificationMethod, RegistrationReason, IdentityMessage, QlickeerEventName vide), ConfigHash inutilisé, auto-affectation domain.setEstablishment(domain.getEstablishment()) dans DenominationDataAccessMapper, blocs commentés partout.
I-11MineurStyle & cohérence : injection par champ @Autowired mélangée à @RequiredArgsConstructor (choisir l'injection par constructeur partout), méthodes en PascalCase (CancelRequest), fautes (retreaving, cupure inconu, colonne total_amount_by_curerency), JUnit 3 (junit:junit:3.8.1) et junit.framework.Assert encore présents, DTOs avec références arrière bidirectionnelles (CashOperationDTO.cashRequest) ⇒ cycles JSON potentiels, enum RequestType mal nommé pour porter un sens d'opération.
I-12MineurHorodatage en double : CashOperationEntity a à la fois les creationDate/modificationDate hérités (gérés par Hibernate) et des createdAt/updatedAt manuels initialisés à LocalDateTime.now(). En garder une seule paire.
13 · Améliorer

Ce qui manque entièrement

  • Sécurité — pas de Spring Security : quiconque atteint le port 8300 peut créer ou annuler des mouvements d'espèces pour n'importe quelle agence. UserProfileEntity existant déjà, brancher OAuth2/JWT (uid tiré du jeton plutôt que du payload) est l'étape naturelle.
  • Le workflow de validation lui-même — les statuts annoncent une chaîne d'approbation (INITIATED → VALIDATED) mais aucun endpoint, aucun acteur, aucun historique. C'est le rôle idéal de Camunda 7 : un processus BPMN Approbation de demande de fonds (tâche responsable d'agence → tâche centre des espèces → rappel du statut vers ce service), ce microservice restant la source de vérité et Camunda l'orchestrateur.
  • Migrations de base — pas de Flyway/Liquibase (cause racine de I-03).
  • Documentation d'API — pas de springdoc-openapi ; le front Angular n'a aucun contrat sur lequel coder.
  • Observabilité — pas d'actuator, pas de sondes santé/readiness, pas de métriques, pas d'ids de corrélation dans les logs.
  • Pagination & filtres — getAll() renvoie tout ; il faut Pageable + filtres (statut, agence, période). JpaSpecificationExecutor est déjà déclaré sur les repositories (sans générique — à typer) mais jamais utilisé.
  • Endpoints de référentiels — le front aura besoin de GET /denominations, GET /currencies, GET /establishments pour construire le formulaire ; les repositories existent, pas les contrôleurs.
  • Référence lisible de demande — le champ domaine uniqueRequestRef existe mais n'est jamais généré ; une référence humaine (ex. CR-2026-000123) est indispensable aux échanges agence / back-office.
  • Audit — qui a changé quel statut, quand : table d'historique ou Hibernate Envers.
  • CI/CD & conteneurisation — pas de Dockerfile, pas de pipeline, pas de barrière qualité (le @SuppressWarnings("java:S119") laisse penser que Sonar est prévu mais non appliqué).
