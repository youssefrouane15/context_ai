1L'essentiel en 30 secondes

La Jira AER_LM-257 demandait le modèle de données des demandes Corporate (alimentation / ramassage de caisse via la Caisse Centrale) et sa gestion dans l'application. Livré :

  • Un modèle relationnel (validé tech lead / PO) dans un schéma PostgreSQL dédié corpo : entête de demande + ventilation normalisée + référentiel des 11 coupures de la Jira, seedé.
  • Une API CRUD complète sur /corpo/cash : création (avec génération de la référence CO-AAAA-NNNNNN et calcul du montant), consultation unitaire et liste, modification, annulation — avec cycle de vie INITIATED / VALIDATED / CANCELLED.
  • Un contrat d'erreur professionnel : 400 / 404 / 409 / 422 au format ProblemDetail, codes API machine + messages métier en français.
  • Le tout sans modifier une seule ligne du périmètre nominal : uniquement des créations, dans l'architecture existante.

2De la Jira au modèle : les choix et leurs raisons

Le modèle retenu

schéma corpo (PostgreSQL) — isolé de public ├── corpo_denomination RÉFÉRENTIEL : les 11 coupures de la Jira, seedées │ (BILLET_200/100/50/20, PIECE_10/5/1, CENTIME_50/20/10/5) ├── corpo_cash_request ENTÊTE : réf, type, statut, client (zones libres), │ montant, urgent, dates création/màj └── corpo_request_denomination VENTILATION : (demande, coupure) → quantité FK CASCADE vers l'entête · FK RESTRICT vers le référentiel · unicité (demande, coupure)

Chaque choix, sa justification

ChoixJustification (à donner telle quelle en démo)
Relationnel plutôt que 11 colonnes platesArbitré avec le tech lead et le PO. La ventilation en lignes normalisées vers un référentiel rend le modèle évolutif : ajouter une coupure demain = une ligne de seed, zéro migration de structure. La contrainte d'unicité (demande, coupure) garantit l'intégrité de la ventilation au niveau BDD, pas seulement applicatif.
Schéma PostgreSQL dédié corpoIsolation nette des deux périmètres : le nominal vit dans public, le corpo dans son schéma, aucune FK croisée. Une seule base, une seule datasource, zéro configuration ajoutée — le ciblage se fait par l'attribut schema des entités JPA. Le jour où le corpo devient un service à part, son schéma part avec lui.
Référentiel de coupures propre au corpoLe référentiel nominal ne couvre pas les centimes (10/20/50 cts, 5 cts) exigés par la Jira, et une FK cross-schema aurait couplé les deux périmètres. Les codes stables (BILLET_200…) servent de contrat d'API : le front envoie un code + une quantité, le serveur résout label et valeur faciale.
Référence lisible CO-AAAA-NNNNNNGénérée serveur sur séquence dédiée — jamais pilotable par le client. Préfixe CO- distinct du nominal, année pour la traçabilité, séquence pour l'unicité garantie par la base.
Statuts INITIATED / VALIDATED / CANCELLEDAlignés sur le nominal pour la cohérence de la plateforme. INITIATED par défaut, annulation exposée (PATCH), VALIDATED prévu en base (contrainte CHECK) mais sans endpoint : la validation arrivera avec le workflow Caisse Centrale — même approche que le nominal.
Zones libres non obligatoiresLa Jira qualifie réf client / ville / adresse / compte de « zones libres » sans exiger d'obligation : nous n'avons inventé aucune contrainte non spécifiée. Les rendre obligatoires est un changement d'une annotation si le métier le demande.
Champs serveur non pilotablesRéférence, statut et dates sont ignorés en entrée (mapper web) : le statut ne change que par le endpoint d'annulation, la référence ne vient que de la séquence. Principe : le client exprime une intention, le serveur détient l'état.

3Architecture : comment le corpo respecte le DDD + hexagonal du projet

Le corpo reproduit exactement la structure en couches du microservice, module par module — aucune entorse, aucune modification de l'existant :

Couche (module)Contenu corpoPrincipe respecté
Domaine
fund-management-domain-core		Agrégat CorpoCashRequest (racine) + CorpoRequestDenomination, value objects d'identité, enums, exceptions métier. Zéro dépendance Spring/JPA — du Java pur.Le cœur métier ne connaît ni le web ni la base : les règles (calcul du montant, défauts, garde du type) vivent dans l'agrégat et sont testables sans framework.
Application service
fund-management-application-service		DTO, mapper web, ports : entrant CorpoDemandUseCase (le contrat offert), sortants CorpoDemandRepository et CorpoDenominationRepository (les besoins exprimés) ; l'implémentation du cas d'usage orchestre.Cœur de l'hexagone : le use case ne dépend que d'interfaces. Il ignore que la persistance est du JPA/Postgres — on pourrait la remplacer sans le toucher.
Adaptateur sortant
fund-management-data-access		Entités JPA (schéma corpo), JpaRepositories, mapper de persistance, CorpoDemandRepositoryImpl / CorpoDenominationRepositoryImpl qui implémentent les ports sortants.Inversion de dépendance : c'est la persistance qui dépend du domaine (elle implémente ses interfaces), jamais l'inverse. Le domaine ne contient aucune annotation JPA.
Adaptateur entrant
fund-management-application		CorpoDemandController (REST, mêmes conventions que le nominal : media type versionné, style CRUD) + CorpoDemandExceptionHandler.Le REST est un simple adaptateur : il traduit HTTP ↔ use case, sans aucune logique métier.

Arguments d'architecture à mettre en avant

  • Le flux traverse les couches dans le bon sens : Controller → port entrant → agrégat (règles) → port sortant → adaptateur JPA. Chaque flèche pointe vers l'intérieur de l'hexagone.
  • Isolation par périmètre : tout le corpo vit dans des sous-paquets corpo dédiés — visible d'un coup d'œil, extractible d'un bloc (vers un module common ou un microservice) le jour venu.
  • Zéro modification de l'existant : ni la classe main, ni les poms, ni une classe du nominal. Les entités/repositories corpo sont placés en sous-paquets des racines déjà scannées par @EntityScan/@EnableJpaRepositories — intégration sans intrusion.
  • Résolution du référentiel par code (clé stable), là où le nominal résout par objet/label — un choix qui évite une classe de bugs de correspondance.

6Qualité d'ingénierie

  • Tests sur toutes les couches : domaine (règles de calcul, défauts, cas d'erreur), mappers (web + persistance, résolution du référentiel), controller (contrat d'erreur testé statut par statut : 400 avec fieldErrors, 404, 409, 422), adaptateur (génération de référence, gardes du cycle de vie, flush).
  • Zéro magic string : statuts, types et flag urgent en enums de bout en bout — colonnes varchar et contrat JSON inchangés, mais toute valeur invalide rejetée dès la désérialisation.
  • MapStruct : mapper web entièrement généré à la compilation, mapper de persistance hybride (mécanique générée, logique de résolution manuelle) — moins de boilerplate, moins de bugs de recopie, sans rien changer au build (processor déjà dans le pom parent).
  • Bug réel trouvé et corrigé par le test manuel : violation de la contrainte d'unicité à l'update (ordre insert/delete d'Hibernate) — diagnostiqué, corrigé (flush), couvert par un test. Bon exemple à raconter : la contrainte BDD a fait son travail.
  • Discipline de périmètre : git status de la PR = uniquement des ajouts ; le nominal est byte-identique.

8Questions anticipées & réponses préparées

« Pourquoi un schéma dédié plutôt que des tables dans public ? »
Isolation des périmètres sans coût : une base, une datasource, zéro config — juste l'attribut schema sur les entités. Droits différenciables plus tard, et extraction facilitée si le corpo devient un service. Le nominal n'est pas pollué.
« Pourquoi un référentiel de coupures séparé du nominal ? »
Le référentiel nominal ne contient pas les centimes exigés par la Jira, et une FK cross-schema aurait couplé les périmètres. La mutualisation reste possible via le futur module common — question ouverte assumée, pas un oubli.
« Le montant d'une alimentation, pourquoi le calculer côté serveur ? »
Un montant d'opération de caisse ne peut pas être déclaratif : il découle de la ventilation. Le serveur fait autorité, et si le front envoie un montant divergent, on le rejette en 422 au lieu de masquer l'incohérence — dans une banque, une divergence de montant se signale, elle ne se corrige pas silencieusement.
« Pourquoi pas d'endpoint de validation (VALIDATED) ? »
Même position que le nominal : la valeur est prévue (contrainte en base), mais le déclencheur relève du workflow Caisse Centrale à venir (Camunda). Exposer un PATCH validate aujourd'hui serait préjuger du processus métier.
« Le nominal ne renvoie que des 500, pourquoi le corpo fait différemment ? »
Le handler est volontairement scopé au controller corpo : le nominal est inchangé. Le corpo sert de prototype du contrat d'erreur cible ; la généralisation au nominal est un chantier identifié, à mener explicitement, pas en effet de bord.
« Pourquoi ne pas avoir mutualisé du code dans un module common tout de suite ? »
Le corpo réutilise le socle existant (BaseId, AggregateRoot, BaseEntity, conventions). Ce qui mériterait common — exceptions génériques, handler global, cycle de vie — n'a de valeur qu'appliqué aux deux périmètres : on le créera au moment de généraliser, avec un contenu réel plutôt qu'un module vide.
« Qu'est-ce qui garantit l'intégrité de la ventilation ? »
Trois niveaux : Bean Validation à l'entrée (code obligatoire, quantité ≥ 0), résolution stricte contre le référentiel (coupure inconnue = rejet, transaction annulée), et contraintes BDD (unicité demande/coupure, CHECK ≥ 0, FK RESTRICT sur le référentiel). Le bug d'unicité trouvé à l'update prouve que ce dernier filet fonctionne.

9Ce qui reste ouvert (transparence)

  • Endpoint / déclencheur de VALIDATED — en attente du processus de validation Caisse Centrale (workflow).
  • isUrgent : enum YES/NO conservé pour la continuité de contrat ; un Boolean serait plus idiomatique — arbitrage tech lead.
  • Champs obligatoires des zones libres — à préciser par le métier si besoin (une annotation par champ).
  • Backlog technique : pagination sur la liste, tests d'intégration JPA (Testcontainers), généralisation du contrat d'erreur au nominal, module common au premier vrai partage.
Message de conclusion pour la démo : « La Jira est livrée dans l'architecture de la plateforme, avec les arbitrages validés ensemble, un contrat d'API strict, et sans toucher un octet de l'existant. Les points ouverts sont identifiés et attendent des décisions métier, pas des correctifs. »
