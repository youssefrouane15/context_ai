1Lecture des notes → décisions actées

Note tech leadTraduction technique retenue
« Dupliquer le schéma tel qu'il est pour corpo »Le schéma Postgres corpo reproduit la structure des tables nominales : cash_request → cash_operation → operation_currency → operation_denomination + référentiels currency/denomination. Les besoins Jira s'ajoutent en colonnes additionnelles (identité client sur cash_request, denomination_code sur denomination) sans altérer la structure. Mapping métier naturel : une demande corpo = 1 cash_request + 1 cash_operation (operation_sens = ALIMENTATION/RAMASSAGE — l'équivalent corpo du SUPPLY/COLLECTION retail) + 1 operation_currency MAD + N operation_denomination.
« Flyway pour la gestion des migrations DB »Flyway activé : la base existante est baselinée V1 (non touchée), le schéma corpo arrive en migration V2 versionnée dans le repo. ddl-auto passe à none : la base est désormais pilotée par les migrations, plus par Hibernate.
« Séparer les deux modèles et couches, package corpo et package retail »Chaque couche a ses deux sous-paquets : le corpo y est déjà, le nominal déménage dans des paquets retail (refactoring IDE automatique, table de correspondance fournie). La classe main du container reste intacte : les nouveaux paquets restent des sous-paquets des racines déjà scannées.
« Module common pour tout ce qui est partagé »Nouveau module Maven fund-management-common avec le socle réellement partagé : BaseId, BaseEntity/AggregateRoot du domaine, BaseEntity JPA. Packages conservés à l'identique → zéro import à changer dans tout le projet (choix assumé pour un diff minimal ; un Move vers des packages common.* reste possible ensuite).
« Suivre l'existant pour les exceptions (pattern BFF) »Différé — en attente des éléments du projet orchestrator-bff. Le handler corpo actuel (ProblemDetail scopé) reste en place provisoirement et sera réaligné sur le pattern BFF dès réception.

4Chantier C — Flyway + migration V2

Intégration from scratch, pas à pas

  1. Dépendances : ajouter les 2 dépendances ci-dessous dans le pom du container (versions gérées par Spring Boot 3.3.1 ; Flyway 10 sépare le support PostgreSQL dans un artefact dédié, d'où la seconde). Reload Maven.
  2. Dossier des migrations : créer fund-management-container/src/main/resources/db/migration/ (clic droit sur resources → New → Directory → taper db/migration). C'est l'emplacement standard scanné par Flyway.
  3. Configuration : remplacer application.yaml par la version ci-dessous. Les points qui comptent : ddl-auto: none (Hibernate ne touche plus la base — c'est Flyway qui pilote), baseline-on-migrate: true + baseline-version: 1 (la base EXISTANTE est marquée V1 sans être modifiée — indispensable sur une base non vide, sinon Flyway refuse de démarrer), schemas: public, corpo.
  4. La migration : créer le fichier V2__create_corpo_schema.sql (respecter le nommage : V + numéro + DEUX underscores + description) avec le contenu ci-dessous.
  5. Premier démarrage : dans les logs, chercher Successfully baselined schema with version: 1 puis Migrating schema "public" to version "2 - create corpo schema". Vérifier dans pgAdmin : table public.flyway_schema_history (2 lignes : baseline + V2) et le schéma corpo avec ses 8 tables + seed.
  6. Règles de vie avec Flyway (à retenir) : un fichier V appliqué ne se modifie JAMAIS (Flyway vérifie les checksums — toute retouche = erreur « Migration checksum mismatch » ; en dev, corriger = DELETE FROM flyway_schema_history WHERE version='2'; + DROP du schéma + relance). Toute évolution future = nouveau fichier V3, V4… Ça vaut aussi pour le retail désormais.

-- ============================================================================
-- V2 — Schema corpo : DUPLICATION du schema retail tel qu'il est (note tech
-- lead), ADAPTE aux nouvelles donnees de la Jira AER_LM-257 :
--   + colonnes identite client sur cash_request (ref client, ville,
--     complement adresse, num compte = zones libres Jira) + unique_request_ref
--   + colonne denomination_code sur denomination (cle stable de l'API)
--   ~ user_profile_id / establishment_id conserves (alignement retail) mais
--     NULLABLES : la Jira identifie par le client, pas par un user/agence
-- La base existante (retail) est baselinee V1 et n'est pas touchee.
-- ============================================================================

DROP SCHEMA IF EXISTS corpo CASCADE;  -- remplace l'ancien modele corpo v1
CREATE SCHEMA corpo;

-- ============ REFERENTIELS (dupliques du retail) ============

CREATE TABLE corpo.currency (
    id                 uuid PRIMARY KEY,
    creation_date      timestamp,
    modification_date  timestamp,
    iso_code           varchar(3) NOT NULL UNIQUE,
    currency_name      varchar(100) NOT NULL
);

CREATE TABLE corpo.denomination (
    id                 uuid PRIMARY KEY,
    creation_date      timestamp,
    modification_date  timestamp,
    item_type          varchar(10) NOT NULL,
    face_value         numeric(10,2) NOT NULL,
    denomination_label varchar(100) NOT NULL,
    denomination_code  varchar(20) NOT NULL UNIQUE,  -- ADDITIONNEL corpo (cle API)
    CONSTRAINT chk_corpo_item_type CHECK (item_type IN ('BANKNOTE', 'COIN'))
);

CREATE TABLE corpo.establishment (
    id                 uuid PRIMARY KEY,
    creation_date      timestamp,
    modification_date  timestamp,
    establishment_code varchar(3) NOT NULL UNIQUE,
    establishment_name varchar(100) NOT NULL
);

CREATE TABLE corpo.user_profile (
    id                 uuid PRIMARY KEY,
    creation_date      timestamp,
    modification_date  timestamp,
    uid                varchar(255),
    profile_code       varchar(50),
    profile_label      varchar(100),
    profile_type       varchar(10)
);

-- ============ STRUCTURE DEMANDE (dupliquee du retail) ============

CREATE TABLE corpo.cash_request (
    id                 uuid PRIMARY KEY,
    creation_date      timestamp,
    modification_date  timestamp,
    branch_code        varchar(100),
    request_status     varchar(30) NOT NULL DEFAULT 'INITIATED',
    -- Alignement retail : FKs conservees, mais NULLABLES (adaptation Jira :
    -- pas de user/etablissement dans une demande corpo, le client remplace)
    user_profile_id    uuid,
    establishment_id   uuid,
    -- ADDITIONNEL corpo : reference lisible + identite client (zones libres)
    unique_request_ref varchar(30) UNIQUE,
    client_ref         varchar(50),
    city               varchar(100),
    address_complement varchar(255),
    account_number     varchar(34),
    CONSTRAINT fk_corpo_req_user FOREIGN KEY (user_profile_id)
        REFERENCES corpo.user_profile(id),
    CONSTRAINT fk_corpo_req_establishment FOREIGN KEY (establishment_id)
        REFERENCES corpo.establishment(id),
    CONSTRAINT chk_corpo_request_status
        CHECK (request_status IN ('INITIATED', 'VALIDATED', 'CANCELLED'))
);

CREATE TABLE corpo.cash_operation (
    id                      uuid PRIMARY KEY,
    creation_date           timestamp,
    modification_date       timestamp,
    cash_request_id         uuid NOT NULL,
    operation_type          varchar(20),
    operation_sens          varchar(50) NOT NULL,   -- ALIMENTATION / RAMASSAGE
    institution_code        integer NOT NULL DEFAULT 1,
    cash_desk_number        varchar(3) NOT NULL DEFAULT '001',
    is_urgent               varchar(3) NOT NULL DEFAULT 'NO',
    target_cash_desk_number varchar(3),
    total_amount            numeric(15,2) NOT NULL DEFAULT 0.00,
    created_at              timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at              timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP,
    operation_status        varchar(30) NOT NULL DEFAULT 'INITIATED',
    establishment           varchar(100),
    CONSTRAINT fk_corpo_op_request FOREIGN KEY (cash_request_id)
        REFERENCES corpo.cash_request(id) ON DELETE CASCADE,
    CONSTRAINT chk_corpo_op_sens CHECK (operation_sens IN ('ALIMENTATION', 'RAMASSAGE')),
    CONSTRAINT chk_corpo_op_urgent CHECK (is_urgent IN ('YES', 'NO'))
);

CREATE TABLE corpo.operation_currency (
    id                        uuid PRIMARY KEY,
    creation_date             timestamp,
    modification_date         timestamp,
    cash_operation_id         uuid,
    currency_id               uuid NOT NULL,
    total_amount_by_curerency numeric(18,2) NOT NULL,  -- nom conserve (retail)
    CONSTRAINT fk_corpo_oc_operation FOREIGN KEY (cash_operation_id)

    CONSTRAINT fk_corpo_oc_currency FOREIGN KEY (currency_id)

CREATE TABLE corpo.operation_denomination (

    modification_date            timestamp,

    CONSTRAINT uq_corpo_od UNIQUE (operation_currency_detail_id, denomination_id),
    CONSTRAINT chk_corpo_od_quantity CHECK (quantity >= 0)

 * la Jira (demande + ventilation) et le schema DUPLIQUE du nominal (note
 * tech lead) : une demande corpo est persistee comme cash_request + UNE
 * cash_operation (sens = ALIMENTATION/RAMASSAGE, urgent, montant total)
 * + UNE ligne operation_currency (MAD) + N operation_denomination.

 * MapStruct reste utilise sur le web mapper.)

@Component
public class CorpoCashRequestPersistenceMapper {

    public static final String CORPO_CURRENCY_ISO = "MAD";

    // ==================== DOMAINE (plat) -> ENTITES (arbre) ====================

    public CorpoCashRequestEntity toPersistence(
            CorpoCashRequest domain,
            Map<String, CorpoDenominationEntity> denominationsByCode,
            CorpoCurrencyEntity madCurrency) {
        if (domain == null) return null;

        CorpoCashRequestEntity entity = new CorpoCashRequestEntity();
        if (domain.getId() != null && domain.getId().getValue() != null) {
            entity.setId(domain.getId().getValue());
        }
        entity.setUniqueRequestRef(domain.getUniqueRequestRef());
        applyHeaderFields(entity, domain);
        entity.setRequestStatus(domain.getRequestStatus() != null
                ? domain.getRequestStatus().name()
                : CorpoRequestStatus.INITIATED.name());
        entity.setOperations(buildOperationTree(domain, entity, denominationsByCode, madCurrency));
        return entity;
    }

    /** Champs d'entete client (creation ET update). */
    public void applyHeaderFields(CorpoCashRequestEntity entity, CorpoCashRequest domain) {
        entity.setClientRef(domain.getClientRef());
        entity.setCity(domain.getCity());
        entity.setAddressComplement(domain.getAddressComplement());
        entity.setAccountNumber(domain.getAccountNumber());
    }

    /** Construit l'operation unique + sa ventilation MAD, rattachees au parent. */
    public List<CorpoCashOperationEntity> buildOperationTree(
            CorpoCashRequest domain,
            CorpoCashRequestEntity parent,
            Map<String, CorpoDenominationEntity> denominationsByCode,
            CorpoCurrencyEntity madCurrency) {

        CorpoCashOperationEntity operation = new CorpoCashOperationEntity();
        operation.setCashRequest(parent);
        operation.setOperationSens(domain.getRequestType() != null
                ? domain.getRequestType().name() : null);
        operation.setIsUrgent(domain.getIsUrgent() != null
                ? domain.getIsUrgent().name() : UrgencyFlag.NO.name());
        operation.setTotalAmount(domain.getAmount());
        operation.setOperationStatus(domain.getRequestStatus() != null
                ? domain.getRequestStatus().name()
                : CorpoRequestStatus.INITIATED.name());

        CorpoOperationCurrencyDetailEntity detail = new CorpoOperationCurrencyDetailEntity();
        detail.setCashOperation(operation);
        if (madCurrency == null) {
            throw new CorpoValidationException(
                    "La devise " + CORPO_CURRENCY_ISO + " est absente du référentiel corpo");
        }
        detail.setCurrency(madCurrency);
        detail.setTotalAmountByCurrency(domain.getAmount());
        detail.setOperationDenominations(
                toDenominationEntities(domain, detail, denominationsByCode));

        List<CorpoOperationCurrencyDetailEntity> details = new ArrayList<>();
        details.add(detail);
        operation.setCurrencyDetails(details);

        List<CorpoCashOperationEntity> operations = new ArrayList<>();
        operations.add(operation);
        return operations;

                                        + "' n'existe pas dans le référentiel corpo");

        existing.getOperations().addAll(
                mapper.buildOperationTree(request, existing, denominationsByCode(), madCurrency()));
        return mapper.toDomain(corpoJpaRepository.save(existing));
    }

    @Override
    @Transactional
    public CorpoCashRequest cancelById(UUID id) {
        CorpoCashRequestEntity existing = findEntityOrThrow(id);
        if (isCancelled(existing)) {
            throw new CorpoConflictException(
                    "La demande " + existing.getUniqueRequestRef() + " est déjà annulée");
        }
        existing.setRequestStatus(CorpoRequestStatus.CANCELLED.name());
        existing.getOperations().forEach(operation ->
                operation.setOperationStatus(CorpoRequestStatus.CANCELLED.name()));
        return mapper.toDomain(corpoJpaRepository.save(existing));
    }

    private CorpoCashRequestEntity findEntityOrThrow(UUID id) {
        return corpoJpaRepository.findByIdWithTree(id)
                .orElseThrow(() -> new CorpoNotFoundException(
                        "La demande corpo " + id + " est introuvable"));
    }

    private boolean isCancelled(CorpoCashRequestEntity entity) {
        return CorpoRequestStatus.CANCELLED.name().equals(entity.getRequestStatus());
    }

    private Map<String, CorpoDenominationEntity> denominationsByCode() {
        return corpoDenominationJpaRepository.findAll().stream()
                .collect(Collectors.toMap(
                        CorpoDenominationEntity::getDenominationCode, d -> d));
    }

    private CorpoCurrencyEntity madCurrency() {
        return corpoCurrencyJpaRepository.findByIsoCode(
                CorpoCashRequestPersistenceMapper.CORPO_CURRENCY_ISO);
    }
}
