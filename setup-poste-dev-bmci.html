Guide d'installation · Poste de travail

Du laptop vierge au parcours crédit
qui tourne en local.

Ce document est votre checklist d'onboarding technique : chaque outil, chaque configuration, chaque piège d'un environnement bancaire (proxy, certificats interceptés, dépôts internes), dans l'ordre où les traiter. Objectif : à la fin, vous déroulez un dossier de crédit complet sur votre machine — front, back, Camunda, Cockpit.

Windows 10/11 (poste banque) Java 17 · Node 20 · Angular Docker · IntelliJ ≈ 1 à 3 jours selon les accès
⚠ Comment lire les valeurs ⟨À CONFIRMER BMCI⟩

Chaque environnement bancaire a ses URLs internes, son proxy, ses noms de dépôts. Tout ce qui est écrit ⟨À CONFIRMER BMCI⟩ est un emplacement à remplir avec la vraie valeur, à demander au tech lead ou à lire dans le wiki d'équipe. Le reste (commandes, mécanismes, ordre des étapes) est universel et directement applicable. Remplissez la fiche du chapitre 10 au fur et à mesure : elle deviendra VOTRE référence.

Activez WSL2 (wsl --install, redémarrage). Docker s'appuie dessus, et vous aurez un vrai shell Linux pour les scripts. Étape 02 · Le passage obligé

Proxy d'entreprise & certificats SSL

C'est LE chapitre qui fait perdre des journées aux nouveaux arrivants. Deux mécanismes distincts, presque toujours présents en banque, et qui cassent tous vos outils tant qu'ils ne sont pas configurés : le proxy sortant et l'interception TLS.

2.1 — Comprendre en 60 secondes

Fig. 1 — Chaque outil doit savoir deux choses : « passe par le proxy » ET « fais confiance au certificat de la banque ». Un outil configuré à moitié donne les erreurs les plus déroutantes du chap. 09.

Conséquence pratique : quand le proxy re-signe le trafic, vos outils voient un certificat inconnu et refusent (self-signed certificate in chain, PKIX path building failed, x509: certificate signed by unknown authority). La solution propre est toujours la même : ajouter le certificat racine de la banque au magasin de confiance de chaque outil — jamais désactiver la vérification SSL.

2.2 — Récupérer les infos (2 valeurs + 1 fichier)

  • L'URL du proxy : http://⟨proxy.bmci.local⟩:⟨8080⟩ — souvent visible dans Windows : Paramètres → Réseau → Proxy, ou fournie par un script PAC. Demandez aussi la liste no-proxy (les domaines internes qui ne passent PAS par le proxy : *.bmci.local, localhost, nexus interne…).
  • Le certificat racine : un fichier bmci-root-ca.crt fourni par l'IT. Sinon, exportez-le vous-même : Chrome → cadenas sur n'importe quel site HTTPS → certificat → chaîne → exporter la racine en Base64 (.CER). Posez-le dans C:\dev\certs\bmci-root-ca.crt.

2.3 — Configurer le socle : les variables d'environnement

PowerShell (persistant, niveau utilisateur)
[Environment]::SetEnvironmentVariable("HTTP_PROXY",  "http://proxy.bmci.local:8080", "User")
[Environment]::SetEnvironmentVariable("HTTPS_PROXY", "http://proxy.bmci.local:8080", "User")
[Environment]::SetEnvironmentVariable("NO_PROXY",
  "localhost,127.0.0.1,.bmci.local,nexus.bmci.local,keycloak.dev.bmci.local", "User")
# Node/npm : faire confiance au certificat de la banque
[Environment]::SetEnvironmentVariable("NODE_EXTRA_CA_CERTS", "C:\dev\certs\bmci-root-ca.crt", "User")
# → fermer/rouvrir le terminal pour prise en compte
⚠ Proxy authentifié (login/mot de passe AD) ?

Si le proxy exige votre compte AD (erreurs 407 Proxy Authentication Required), ne mettez jamais votre mot de passe dans une variable d'environnement ou un fichier de config. Deux options propres : ① l'entreprise fournit souvent un proxy local non authentifié type Px / Cntlm qui relaie avec votre session Windows — demandez la pratique en vigueur ⟨À CONFIRMER BMCI⟩ ; ② certains proxys authentifient par machine/IP et le 407 n'apparaît jamais. Notez la réponse dans la fiche du chap. 10.

2.4 — Java : importer le certificat dans le truststore

Maven, l'orchestrateur, les workers — tout ce qui est Java lit le magasin cacerts du JDK :

PowerShell (admin) — à refaire à chaque nouveau JDK installé
keytool -importcert -trustcacerts -cacerts `
  -alias bmci-root-ca `
  -file C:\dev\certs\bmci-root-ca.crt -noprompt
# mot de passe par défaut du magasin : changeit
# Vérifier :
keytool -list -cacerts -alias bmci-root-ca

2.5 — Git, npm, Docker : le tour des outils

Git — ~/.gitconfig
[http]
    proxy = http://proxy.bmci.local:8080
    sslCAInfo = C:/dev/certs/bmci-root-ca.crt
# Si les dépôts Git sont INTERNES (GitHub Enterprise en .bmci.local),
# le proxy est inutile pour eux :
[http "https://github.bmci.local"]
    proxy = ""
npm — exécuter une fois
npm config set proxy       http://proxy.bmci.local:8080
npm config set https-proxy http://proxy.bmci.local:8080
# Le certificat est déjà géré par NODE_EXTRA_CA_CERTS (2.3).
# JAMAIS ceci en revanche :  npm config set strict-ssl false   ← interdit
Docker Desktop / Rancher Desktop
1) Proxy : Settings → Resources → Proxies → renseigner HTTP/HTTPS/No-proxy
2) Certificat (pour docker pull) : Settings → Docker Engine, ou copier le .crt dans
   /etc/docker/certs.d/ du backend WSL — la logithèque interne a souvent
   une procédure toute faite ⟨À CONFIRMER BMCI⟩
3) Vérif : docker pull eclipse-temurin:17-jre-alpine

Étape 03 · Code source

Git & clonage des dépôts

3.1 — Identité et authentification

Configuration d'identité (une fois)
git config --global user.name  "Youssef ⟨Nom⟩"
git config --global user.email "y.⟨nom⟩@bmci.ma"    # l'email du compte GitHub d'entreprise
git config --global init.defaultBranch main
git config --global pull.rebase true                # évite les merge commits parasites

Pour l'authentification vers GitHub : en entreprise derrière proxy, HTTPS + PAT (Personal Access Token) est le chemin le plus fiable — le port SSH (22) est souvent bloqué. GitHub → Settings → Developer settings → Fine-grained token, portée limitée à l'organisation, expiration 90 j. Windows le mémorise via le Git Credential Manager (installé avec Git for Windows) : le token n'est saisi qu'une fois.

ℹ Si l'entreprise impose SSH quand même

GitHub accepte SSH sur le port 443 : dans ~/.ssh/config, ciblez Host github.com → HostName ssh.github.com, Port 443 (+ ProxyCommand type connect.exe si le proxy l'exige). Mais validez d'abord la pratique de l'équipe ⟨À CONFIRMER BMCI⟩ — ne vous battez pas seul contre le réseau.

3.2 — Cloner tout le domaine en une commande

C:\dev\credit\clone-all.ps1
$org   = "⟨banque⟩"          # organisation GitHub À CONFIRMER
$repos = @(
  "credit-orchestrator", "credit-scoring-svc", "credit-kyc-svc",
  "credit-contrat-svc", "credit-notif-svc",
  "shell-banque", "mf-espace-client", "mf-corbeille",
  "credit-platform-config"
)
foreach ($r in $repos) {
  if (-not (Test-Path $r)) { git clone "https://github.com/$org/$r.git" }
  else { Write-Host "$r déjà présent, skip" }
}

Convention de rangement : tout sous C:\dev\credit\, un dossier par repo, jamais dans OneDrive/Documents (la synchro cloud + node_modules = poste à genoux, et c'est souvent interdit par la politique interne de toute façon).

3.3 — Le premier réflexe dans chaque repo

Ouvrez le README.md et cherchez trois choses : la version Java/Node attendue, le fichier d'exemple d'environnement (.env.example, application-local.yml.example…) à copier-renommer, et la commande « ça tourne » de l'équipe. Si un de ces trois éléments manque, c'est votre première contribution toute trouvée — un README amélioré par le regard neuf d'un arrivant vaut de l'or.

Étape 04 · Dépendances

Maven, Nexus & npm interne

En banque, les dépendances ne viennent presque jamais d'Internet en direct : elles passent par un miroir interne (Nexus ou Artifactory) qui cache, scanne et filtre. Deux fichiers à configurer, et tout le reste suit.

4.1 — Maven : le settings.xml

C:\Users\⟨vous⟩\.m2\settings.xml
<settings>
  <!-- ① Tout Maven passe par le miroir interne -->
  <mirrors>
    <mirror>
      <id>nexus-bmci</id>
      <mirrorOf>*</mirrorOf>
      <url>https://nexus.bmci.local/repository/maven-public/</url>  <!-- ⟨À CONFIRMER⟩ -->
    </mirror>
  </mirrors>

  <!-- ② Identifiants du miroir (souvent le compte AD ou un token Nexus) -->
  <servers>
    <server>
      <id>nexus-bmci</id>
      <username>⟨login⟩</username>
      <password>⟨token — voir encadré ci-dessous⟩</password>
    </server>
  </servers>

  <!-- ③ Proxy : UNIQUEMENT si Nexus est lui-même derrière le proxy.
       Si Nexus est interne (cas normal), pas de section proxy,
       ou avec nonProxyHosts couvrant *.bmci.local -->
</settings>
⚠ Jamais de mot de passe AD en clair dans settings.xml

Utilisez soit un token utilisateur Nexus (Nexus → votre profil → User Token), soit le chiffrement Maven : mvn --encrypt-master-password puis mvn --encrypt-password et collez la valeur {chiffrée}. Le settings.xml se retrouve vite dans des sauvegardes, des captures d'écran, des tickets…

Vérification : dans credit-orchestrator, lancez mvn dependency:resolve. Le téléchargement doit défiler avec des URLs en nexus.bmci.local — si vous voyez repo.maven.apache.org, le miroir n'est pas pris en compte.

4.2 — npm : le registre interne

Fichier .npmrc — deux niveaux
# ~/.npmrc (global, votre poste)
registry=https://nexus.bmci.local/repository/npm-public/        # ⟨À CONFIRMER⟩
# auth vers Nexus, générée par :  npm login --registry=...
//nexus.bmci.local/repository/npm-public/:_authToken=⟨token⟩

# Dans les repos front, un .npmrc DE PROJET (commité) référence souvent
# le scope interne du design system :
@banque:registry=https://nexus.bmci.local/repository/npm-interne/

Vérification : dans mf-corbeille, npm ci doit passer, y compris le paquet interne @banque/ds-angular — c'est lui qui prouve que le registre à scope fonctionne.

Étape 05 · IDE

IntelliJ IDEA, aux petits oignons

5.1 — Les réglages qui comptent, dans l'ordre

  1. Proxy de l'IDE : Settings → Appearance & Behavior → System Settings → HTTP Proxy → Manual, avec l'exception *.bmci.local. (Sert aux plugins, à la doc, aux SDK — pas à Maven.)
  2. Certificat : Settings → Tools → Server Certificates → ajouter bmci-root-ca.crt (« Accept non-trusted certificates » doit rester décoché).
  3. JDK : File → Project Structure → SDKs → ajouter le Temurin 17 installé. Project SDK = 17, language level = 17.
  4. Maven : Settings → Build Tools → Maven → vérifier que User settings file pointe bien votre settings.xml du chap. 04 (piège classique : IntelliJ utilise son Maven embarqué avec un settings vide).
  5. Lombok : plugin installé (bundled sur les versions récentes) + Settings → Build → Compiler → Annotation Processors → Enable annotation processing. Sans ça, tout le code @RequiredArgsConstructor est rouge.
  6. Plugins utiles : SonarLint (les mêmes règles que la CI, en local), CheckStyle-IDEA si l'équipe en a un, .ignore. Le style de code partagé de l'équipe s'importe via Settings → Editor → Code Style → ⚙ → Import (fichier dans le wiki ⟨À CONFIRMER BMCI⟩).

5.2 — Ouvrir le back proprement

File → Open → sélectionnez le pom.xml de credit-orchestrator → « Open as Project ». Attendez la fin de l'indexation + résolution Maven (barre du bas). Faites pareil pour un worker (credit-contrat-svc) — soit dans une seconde fenêtre, soit attachés au même projet (File → New → Module from Existing Sources) si vous préférez tout dans une fenêtre.

5.3 — Les Run Configurations à créer

NomType & réglages
orchestrator [local]Spring Boot → classe main → Active profiles : local → Environment : rien (tout est dans application-local.yml)
contrat-svc [local]Idem, profil local — pointe base-url: http://localhost:8080/engine-rest
tests processusJUnit → all in package ma.banque.credit — votre bouton « le BPMN est-il cassé ? »
stack docker(optionnel) Run config « Docker Compose » sur le docker-compose.yml — ou gardez ça dans le terminal

Astuce : une Compound configuration « TOUT le back » qui lance orchestrator + les workers d'un clic, une fois que chaque config individuelle fonctionne.

Étape 06 · Back-end

Lancer le back en local

Ordre de démarrage logique : d'abord les dépendances d'infrastructure (base + Keycloak, en Docker), puis l'orchestrateur, puis les workers. À chaque étape, une vérification avant de passer à la suivante.

Fig. 2 — La séquence de démarrage. En cas de problème, on sait toujours quelle couche incriminer : celle qu'on vient d'ajouter.

6.1 — ① L'infra locale

Terminal, dans credit-orchestrator/
docker compose up -d postgres keycloak
docker compose ps          # les 2 services doivent être "healthy"
# Keycloak (realm de dev importé automatiquement) : http://localhost:8081
#   → comptes de test : salma.test / karim.analyste ⟨mots de passe : voir wiki⟩
ℹ Derrière le proxy, premier lancement

Le premier docker compose up télécharge les images — c'est le vrai test de votre config Docker du chap. 2.5. Erreur x509 ou timeout → retournez-y, rien d'autre ne fonctionnera avant.

6.2 — ② et ③ L'orchestrateur

Lancez la run config orchestrator [local]. Dans les logs de démarrage, trois lignes valident tout :

... ENGINE-07015 Detected @EnableProcessApplication ...
... ENGINE-08023 Deployment summary: credit-express.bpmn, niveau-validation.dmn   ← le BPMN est déployé
... Tomcat started on port 8080

Puis ouvrez http://localhost:8080/camunda (admin/admin en local) → Cockpit → le processus credit-express est là, avec son diagramme. Screenshot mental : c'est votre « ça marche » du back.

6.3 — ④ et ⑤ Les workers

Lancez contrat-svc [local] (et les autres selon besoin). La ligne à guetter dans leurs logs :

... TASK/CLIENT-01007 Subscribed to topic 'credit.deblocage-fonds' ...

Au quotidien, inutile de lancer les 4 workers : démarrez seulement ceux du sujet du jour. Un topic sans worker n'est pas une erreur — les jobs s'accumulent tranquillement et seront traités quand le worker démarrera. (C'est même un excellent exercice : démarrez un dossier sans le worker scoring, regardez le jeton attendre dans Cockpit, puis lancez le worker et voyez-le partir.)

6.4 — Voir la base (optionnel mais instructif)

DBeaver → PostgreSQL localhost:5432/camunda_credit. Les tables qui racontent tout : ACT_RU_TASK (les User Tasks en attente), ACT_RU_VARIABLE (le sac à dos des jetons), ACT_RU_JOB (timers et async), ACT_RU_INCIDENT. En lecture seulement — on ne modifie jamais ces tables à la main, même en local, sinon on apprend de mauvais réflexes.

Étape 07 · Front-end

Lancer le front : shell + remotes

7.1 — La topologie locale

Trois serveurs de dev Angular, trois ports — le shell charge les remotes locaux via sa config d'environnement :

AppliPort localCommande
shell-banque4200npm ci && npm start
mf-espace-client4201npm ci && npm start
mf-corbeille4202npm ci && npm start

La config environment.local.ts du shell pointe http://localhost:4201/remoteEntry.js et 4202 — vérifiez-la, c'est le câblage Module Federation local. Vous ne travaillez généralement que sur UN remote : de nombreuses équipes configurent le shell pour charger les autres remotes depuis l'environnement de dev partagé, pour ne lancer que 2 serveurs au lieu de 3 ⟨pratique À CONFIRMER BMCI⟩.

7.2 — Le proxy de dev Angular (à ne pas confondre avec celui du chap. 02 !)

Le navigateur appelle /api/** sur le port 4200 ; il faut relayer vers l'orchestrateur local. C'est le rôle du proxy.conf.json d'Angular — un proxy de développement, qui n'a rien à voir avec le proxy d'entreprise :

shell-banque/proxy.conf.json
{
  "/api":    { "target": "http://localhost:8080", "secure": false },
  "/auth":   { "target": "http://localhost:8081", "secure": false }
}

(Référencé dans angular.json → serve → options → proxyConfig, généralement déjà en place.) Résultat : même origine pour le navigateur, zéro problème CORS en local.

7.3 — Se connecter

Ouvrez http://localhost:4200 → redirection Keycloak local → connectez-vous en karim.analyste → menu « Corbeille crédit » : le shell charge mf-corbeille depuis le port 4202. Si la corbeille est vide, c'est normal : aucun dossier n'existe encore — on règle ça au chapitre suivant.

💡 Deux profils, deux navigateurs

Gardez Chrome = Karim (analyste) et une fenêtre privée ou Edge = Salma (cliente) connectés en parallèle : indispensable pour dérouler le parcours complet sans se déconnecter/reconnecter sans arrêt.

Étape 09 · SOS

Dépannage : les 12 erreurs classiques

Gardez ce chapitre sous le coude : ce sont les erreurs que rencontre chaque nouvel arrivant en environnement bancaire, avec leur cause réelle et le fix. Le réflexe général : lisez le nom de l'erreur, pas seulement sa couleur — 90 % du diagnostic est dans le message.

Symptôme (le message exact)Cause & remède
407 Proxy Authentication Required
(mvn, npm, git…)		Le proxy exige votre compte AD. → Solution d'entreprise (Px/Cntlm ou proxy par IP), chap. 2.3. Jamais le mot de passe en clair dans une config.
PKIX path building failed: unable to find valid certification path
(Maven, appli Java)		Le JDK ne connaît pas le certificat de la banque. → keytool -importcert (chap. 2.4). À refaire pour chaque JDK installé — c'est par JDK, pas par machine.
SELF_SIGNED_CERT_IN_CHAIN ou UNABLE_TO_GET_ISSUER_CERT_LOCALLY
(npm, ng)		Node ne connaît pas le certificat. → NODE_EXTRA_CA_CERTS (chap. 2.3), puis rouvrir le terminal. Ne JAMAIS répondre par strict-ssl=false.
x509: certificate signed by unknown authority
(docker pull)		Le démon Docker ne connaît pas le certificat. → chap. 2.5 point 2. C'est le démon qu'il faut configurer, pas votre shell.
ECONNRESET / ETIMEDOUT aléatoires (npm)Proxy configuré à moitié (https-proxy manquant) ou NO_PROXY qui n'exclut pas Nexus interne. → relire 2.3 + 4.2, vérifier avec npm config list.
Maven télécharge depuis repo.maven.apache.org au lieu de NexusIntelliJ utilise son Maven embarqué avec un autre settings.xml. → Settings → Build Tools → Maven → User settings file (chap. 5.1 point 4).
Tout est rouge dans IntelliJ : cannot resolve symbol 'log', constructeurs introuvablesLombok : annotation processing désactivé. → chap. 5.1 point 5, puis Build → Rebuild Project.
Port 8080 was already in useUn autre orchestrateur (ou Tomcat, ou un conteneur) tient le port. → netstat -ano | findstr :8080 puis tuer le PID, ou changer server.port du profil local.
Au démarrage : delegateExpression ... Unknown property used in expression: ${...Delegate}Le bean Spring n'existe pas (faute de frappe dans le BPMN, ou classe non scannée). → comparer le nom dans le Modeler et le @Component("..."). C'est LE bug Camunda du débutant.
Le worker tourne mais ne prend aucun job3 suspects dans l'ordre : ① topic mal orthographié (comparer BPMN ↔ @ExternalTaskSubscription) ; ② base-url du client qui ne pointe pas localhost:8080/engine-rest ; ③ le jeton n'est pas encore arrivé sur la tâche (regarder Cockpit).
Le shell affiche une page blanche sur la route du remote / erreur remoteEntry.js dans la consoleLe remote n'est pas démarré, ou l'URL dans environment.local.ts ne correspond pas au port réel. F12 → onglet Network → chercher remoteEntry.js : son statut dit tout.
401/403 sur /api/corbeille alors que le login a marchéLe JWT ne porte pas le bon groupe (compte de test mal configuré dans le realm local) ou le mapping groupes→candidate groups diffère. → décoder le token sur jwt.io (celui de DEV local uniquement !) et comparer la claim groups.
ℹ La méthode quand rien ne colle

Isolez la couche : la commande passe-t-elle hors VPN / sur un autre réseau ? (→ problème proxy/cert). Passe-t-elle en curl brut vers la même URL ? (→ problème de l'outil, pas du réseau). Passe-t-elle chez un collègue ? (→ votre poste). Trois questions, et le coupable est cerné — notez la solution dans le wiki d'équipe pour le suivant.
