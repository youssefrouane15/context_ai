3Cas nominaux

CN-01POST/corpo/cash201Création d'une demande ALIMENTATION avec ventilation (le cas de référence)

Règle vérifiée : Montant CALCULÉ serveur (200×10 + 100×5 = 2500.00), référence lisible générée CO-AAAA-NNNNNN, statut INITIATED, défauts serveur appliqués (institution_code=1, cash_desk_number='001').

Payload (Body JSON)
{
  "branchCode": "AG-CASA-01",
  "establishment": "001",
  "user": "j43709",
  "clientRef": "CORP-00042",
  "city": "Casablanca",
  "addressComplement": "Zone industrielle Ain Sebaa, lot 12",
  "accountNumber": "011780000012345678901234",
  "operations": [
    {
      "operationSens": "ALIMENTATION",
      "isUrgent": "YES",
      "currencyDetails": [
        {
          "currency": { "isoCode": "MAD" },
          "operationDenominations": [
            { "denomination": { "denominationCode": "BILLET_200" }, "quantity": 10 },
            { "denomination": { "denominationCode": "BILLET_100" }, "quantity": 5 }
          ]
        }
      ]
    }
  ]
}

Réponse attendue

  • Statut 201 ; id (UUID) et uniqueRequestRef = CO-2026-000001 (ou N suivant) présents
  • requestStatus = INITIATED ; l'opération porte totalAmount = 2500.00 et operationStatus = INITIATED
  • Chaque coupure revient enrichie : denominationLabel (« Billet de 200 DH »), faceValue, itemType
  • Noter l'id retourné → utilisé par CN-04/CN-06/CN-07

Attendu en base de données

TableCe que tu dois voir
corpo.cash_request1 ligne : unique_request_ref = CO-2026-000001, request_status = INITIATED, client_ref = CORP-00042, city = Casablanca, account_number rempli, branch_code = AG-CASA-01 ; establishment_id / user_profile_id remplis si le seed optionnel a été inséré, sinon NULL
corpo.cash_operation1 ligne : operation_sens = ALIMENTATION, is_urgent = YES, total_amount = 2500.00, operation_status = INITIATED, institution_code = 1, cash_desk_number = 001
corpo.operation_currency1 ligne : currency_id → MAD, total_amount_by_curerency = 2500.00
corpo.operation_denomination2 lignes : (BILLET_200, quantity 10) et (BILLET_100, quantity 5)
Vérification SQL (pgAdmin)
SELECT cr.unique_request_ref, cr.request_status, cr.client_ref,
       op.operation_sens, op.is_urgent, op.total_amount, op.institution_code, op.cash_desk_number,
       oc.total_amount_by_curerency, d.denomination_code, od.quantity
FROM corpo.cash_request cr
JOIN corpo.cash_operation op        ON op.cash_request_id = cr.id
JOIN corpo.operation_currency oc    ON oc.cash_operation_id = op.id
JOIN corpo.operation_denomination od ON od.operation_currency_detail_id = oc.id
JOIN corpo.denomination d           ON d.id = od.denomination_id
ORDER BY cr.creation_date DESC;

Règle vérifiée : L'update remplace l'arbre d'opérations (clear + flush + re-add) et met à jour l'entête client ; unique_request_ref et statut sont immuables par ce canal.

        {
          "currency": { "isoCode": "MAD" },
          "operationDenominations": [
            { "denomination": { "denominationCode": "BILLET_100" }, "quantity": 20 }
          ]
        }
      ]
    }
  ]
}

Réponse attendue

  • Sur l'id du CN-01 ; statut 200
  • uniqueRequestRef INCHANGÉE (toujours CO-2026-000001) — preuve d'immutabilité
  • isUrgent = NO, totalAmount = 2000.00, une seule coupure BILLET_100 × 20

Attendu en base de données

TableCe que tu dois voir

Vérification SQL (pgAdmin)
-- Il ne doit rester qu'UNE opération et UNE ligne de coupure pour cette demande
SELECT (SELECT count(*) FROM corpo.cash_operation op WHERE op.cash_request_id = cr.id)  AS nb_operations,
       (SELECT count(*) FROM corpo.operation_denomination od
         JOIN corpo.operation_currency oc ON od.operation_currency_detail_id = oc.id
         JOIN corpo.cash_operation op2 ON oc.cash_operation_id = op2.id
        WHERE op2.cash_request_id = cr.id) AS nb_lignes_coupures,
       cr.unique_request_ref, cr.client_ref
FROM corpo.cash_request cr WHERE cr.unique_request_ref = 'CO-2026-000001';

4Cas d'erreur & edge cases

Vérification transverse à faire après CHAQUE cas d'erreur : aucune écriture ne doit s'être produite (transaction rollback). Garde ce compteur sous la main et vérifie qu'il ne bouge pas : SELECT (SELECT count(*) FROM corpo.cash_request) AS demandes, (SELECT count(*) FROM corpo.cash_operation) AS operations, (SELECT count(*) FROM corpo.operation_denomination) AS lignes;
ER-01POST/corpo/cash400Bean Validation : operations manquantes

Règle vérifiée : @NotEmpty sur operations — une demande sans opération est refusée AVANT toute logique métier.

Payload (Body JSON)
{ "user": "j43709", "clientRef": "CORP-00500" }

Réponse attendue

  • Statut 400, code = CORPO_BAD_REQUEST
  • fieldErrors.operations présent avec le message de contrainte
