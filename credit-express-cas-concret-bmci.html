« Crédit Express » — une vraie application bancaire,
du clic du client au pipeline GitHub.

Ce document raconte une seule histoire : celle d'un parcours de crédit conso digital, construit avec Camunda 7, des microservices Spring Boot, un front Angular en micro-frontends, et déployé sur Kubernetes via GitHub Actions. Tout est fictif mais réaliste : les noms, les repos, les YAML, les incidents. L'objectif : que vous puissiez vous projeter dans chaque couche.

Camunda 7.21 4 microservices + 1 orchestrateur Shell + 2 remotes Angular GitHub Actions + GitOps Kubernetes

Le scénario

Salma, cliente de la banque depuis 6 ans, veut financer des travaux : 60 000 DH sur 36 mois. Elle fait sa demande un mardi à 14h12 depuis son espace client web. Voici les personnages qui vont toucher son dossier :

👩🏽 Salma — la cliente

Elle remplit un formulaire, signe électroniquement, et suit son dossier comme un colis. Elle ne sait pas ce qu'est Camunda, et c'est très bien comme ça.

👨🏻‍💼 Karim — l'analyste crédit

Il travaille dans sa « corbeille » : une liste de dossiers à instruire, avec des délais. Il accorde, refuse, ou demande un complément. Son outil : le micro-frontend mf-corbeille.

👩🏻‍💻 Vous — l'équipe « Parcours Crédit »

Une squad : 3 devs back, 2 devs front, 1 PO, 1 ops. Elle possède tout le parcours : le BPMN, les services, les deux remotes front, et ses pipelines. C'est le principe « you build it, you run it ».

🖥️ La prod — un cluster Kubernetes

Trois environnements (dev, uat, prod), un registre d'images, et des déploiements déclenchés par Git. Personne ne fait de kubectl apply à la main sur la prod.

ℹ Comment lire ce dossier

Les chapitres 01–03 posent la carte (architecture, front, back). Le chapitre 04 est le cœur : on suit le dossier de Salma minute par minute à travers chaque composant. Les chapitres 05–08 montrent comment tout ça vit : conteneurs, pipelines GitHub, promotion entre environnements, supervision. L'architecture & les dépôts GitHub

Règle d'organisation : un composant déployable = un dépôt GitHub = un pipeline. La squad possède 8 dépôts, tous dans l'organisation GitHub de la banque, préfixés par le domaine credit-.

💡 Pourquoi pas un monorepo ?

Les deux marchent. Ici, le multi-repo colle au principe « un déployable = un cycle de vie » : le worker notification peut être livré 4 fois par jour sans toucher l'orchestrateur. Le point important n'est pas le nombre de repos, c'est que la squad possède l'ensemble — y compris le BPMN et le dépôt GitOps. Si votre organisation préfère un monorepo avec des pipelines par dossier, tout ce document reste valable.

1.3 — Qui parle à qui — et qui ne parle JAMAIS à qui

  • Le front ne connaît qu'une URL : la gateway. Il ne sait pas que Camunda existe (il appelle /api/demandes, /api/corbeille — la façade).
  • Les workers ne se connaissent pas entre eux et ne sont appelés par personne : ils tirent leur travail depuis /engine-rest (réseau interne du cluster uniquement).
  • Seuls credit-scoring-svc et credit-contrat-svc parlent au core banking — l'orchestrateur, lui, n'a aucun accès aux systèmes centraux. Chacun son périmètre réseau, les flux à ouvrir restent courts à documenter pour la sécurité.
  • Personne ne touche la base camunda_credit à part l'orchestrateur. Les workers n'y ont même pas de compte.
Chapitre 02 · Front

Le front : un shell, deux remotes

Le principe micro-frontend en une phrase simple : le site de la banque est un puzzle. Le « shell » est le cadre du puzzle (menu, login, routing) ; chaque équipe fournit ses pièces (« remotes »), développées et déployées indépendamment, assemblées dans le navigateur au moment du chargement.

2.1 — Le contrat entre le shell et les remotes

Le shell (possédé par l'équipe « socle digital ») publie un contrat minimal : chaque remote expose ses routes via Module Federation, et reçoit du shell le token Keycloak et le thème. Côté squad crédit, la configuration :

mf-corbeille/federation.config.js
module.exports = {
  name: 'mfCorbeille',
  exposes: { './routes': './src/app/corbeille.routes.ts' },
  shared: ['@angular/core', '@angular/common', '@angular/router'],
};
shell-banque — déclaration de la pièce de puzzle
{
  path: 'credit/corbeille',
  canActivate: [authGuard, roleGuard(['analystes-credit', 'comite-credit'])],
  loadChildren: () => loadRemoteModule({
    remoteEntry: environment.remotes.mfCorbeille,   // URL injectée par env
    exposedModule: './routes',
  }).then(m => m.routes),
}
💡 Le point clé pour la suite (CI/CD)

L'URL remoteEntry n'est pas codée en dur : elle vient d'une config par environnement. En uat, le shell charge https://uat.banque.ma/mf/corbeille/remoteEntry.js ; en prod, la version prod. Résultat : déployer une nouvelle version de la corbeille ne nécessite AUCUN redéploiement du shell — on remplace juste l'image nginx du remote. C'est ça, l'autonomie micro-frontend, et c'est ce que le pipeline du chap. 06 exploite.

2.2 — Ce que voit Salma vs ce que voit Karim

mf-espace-client (Salma)

Écran 1 : formulaire de demande (montant, durée, revenus) → POST /api/demandes.
Écran 2 : suivi « type colis » : Reçue ✓ → Analysée ✓ → En attente de votre signature ⏳, alimenté par GET /api/demandes/CR-2026-08812/timeline (l'historique Camunda, traduit en langage client).

mf-corbeille (Karim)

Écran 1 : la corbeille — tableau des tâches de ses groupes, tri par échéance SLA, badge rouge si dépassée → GET /api/corbeille.
Écran 2 : le dossier — variables du processus + données métier enrichies + formulaire de décision, choisi dynamiquement par la formKey de la tâche.

Les deux remotes ne partagent aucun code métier entre eux — seulement le design system de la banque (une librairie npm interne versionnée, ex. @banque/ds-angular).

Chapitre 03 · Back

Le back : l'orchestrateur et ses workers

3.1 — L'orchestrateur : le chef de gare

credit-orchestrator est un Spring Boot classique qui contient trois choses et rien d'autre :

  • Le moteur Camunda + les processus : credit-express.bpmn et niveau-validation.dmn dans src/main/resources/processes/, auto-déployés au démarrage.
  • La façade REST /api/** : la seule porte d'entrée du front. Elle traduit « métier » ↔ « moteur » : POST /api/demandes → startProcessInstanceByKey, GET /api/corbeille → TaskQuery filtrée par les groupes du JWT, etc.
  • Quelques delegates légers : préparation de variables, mapping — jamais d'appel au core banking (ça, c'est le rôle des workers).

3.2 — Les workers : des artisans spécialisés

Chaque worker est un mini Spring Boot sans moteur Camunda — juste le client External Task (~2 Mo) et son métier. Exemple type, le déblocage de fonds :

credit-contrat-svc/DeblocageFondsWorker.java
@Component
@ExternalTaskSubscription(topicName = "credit.deblocage-fonds")
@RequiredArgsConstructor
public class DeblocageFondsWorker implements ExternalTaskHandler {

    private final CoreBankingClient coreBanking;
    private final VirementDejaEmisRepository dejaEmis;   // garde-fou idempotence

    @Override
    public void execute(ExternalTask task, ExternalTaskService svc) {
        String dossier = task.getBusinessKey();          // "CR-2026-08812"
        Long montant   = task.getVariable("montantDh");
        try {
            // Idempotence : si un retry rejoue la tâche, on ne vire pas 2 fois
            if (!dejaEmis.existsByDossier(dossier)) {
                String refVirement = coreBanking.emettreVirement(dossier, montant);
                dejaEmis.save(new VirementEmis(dossier, refVirement));
            }
            svc.complete(task);
        } catch (CompteClotureException e) {
            svc.handleBpmnError(task, "COMPTE_CLOTURE", e.getMessage()); // chemin métier
        } catch (Exception e) {
            int retries = task.getRetries() == null ? 3 : task.getRetries() - 1;
            svc.handleFailure(task, e.getMessage(), "", retries, 120_000L); // technique
        }
    }
}
⚠ La règle des workers, à afficher au mur

Un worker fait une chose, de façon idempotente, et classe chaque échec en technique (handleFailure → retry) ou métier (handleBpmnError → chemin du diagramme). Il ne connaît ni le processus global, ni les autres workers. C'est ce qui rend l'ensemble simple à raisonner : la complexité de l'enchaînement vit dans UN endroit — le BPMN.

Chapitre 04 · Le cœur du dossier

On suit le dossier de Salma, minute par minute

Mardi, 14h12. Salma clique sur « Envoyer ma demande ». Voici exactement ce qui se passe dans chaque composant, avec les vrais appels. Si vous ne lisez qu'un chapitre, c'est celui-là.

14h12:04 — La demande naît

Fig. 3 — Grâce au « async before », la réponse HTTP ne dépend pas du scoring ni du KYC : le dossier est en sécurité en base, la suite se fera en tâche de fond.

14h12:05 → 14h12:11 — Les machines travaillent

Le Job Executor de l'orchestrateur exécute « Préparer le dossier », puis le jeton se duplique au gateway parallèle : deux External Tasks sont publiées, topics credit.scoring et credit.kyc.

Fig. 4 — Six secondes plus tard, le dossier attend dans la corbeille des analystes. Aucun humain n'a rien fait, aucun thread n'attend : l'état dort en base.

16h47 — Karim ouvre sa corbeille

Le remote mf-corbeille appelle GET /api/corbeille. La façade lit les groupes du JWT de Karim (analystes-credit), interroge le moteur, enrichit chaque tâche avec les données métier, et renvoie :

Réponse de GET /api/corbeille (extrait)
[{
  "taskId": "8842a1...",
  "formKey": "instruction-analyste",
  "businessKey": "CR-2026-08812",
  "due": "2026-07-09T14:12:11Z",
  "slaStatut": "OK",
  "dossier": { "client": "Salma B.", "anciennete": "6 ans",
               "montantDh": 60000, "dureeMois": 36,
               "score": 712, "tauxEndettement": 0.31 }
}]

Karim clique. Le registre de formulaires du remote mappe instruction-analyste → le composant Angular du même nom. Il examine, puis choisit « Demander un complément » : il veut le justificatif des revenus locatifs. → POST /api/corbeille/8842a1/decision {decision:"COMPLEMENT", message:"..."}. Le jeton repart : gateway « décision ? » → branche COMPLEMENT → nouvelle User Task, mais cette fois candidateGroup = clients avec formKey = complement-client : elle apparaît chez Salma, dans son espace client, avec une notification (le worker notif est appelé au passage par un listener).

Mercredi 09h30 — Salma répond, la boucle se referme

Salma téléverse son justificatif (stocké dans la GED via la façade — pas dans les variables Camunda, on n'y met que l'ID du document). Sa tâche est complétée, le jeton reboucle sur « Instruction analyste ». Karim revoit le dossier à 10h05 : ACCORD. Le jeton file vers la génération du contrat (worker credit-contrat-svc), puis s'endort sur l'event-based gateway : signature ou 7 jours.

Mercredi 10h11 — La signature réveille le dossier

Fig. 5 — Fin du parcours : 20 heures de vie, 2 humains, 4 microservices, zéro cron, zéro colonne de statut maison. À 10h12, le téléphone de Salma vibre.

💡 Le test à se poser sur votre propre refonte

Reprenez ce chapitre et demandez-vous, pour un parcours BMCI réel : « où serait stocké l'état ? qui gérerait le délai de 48h ? comment le métier verrait-il où en est le dossier ? que se passerait-il si le pod redémarre à 14h12:06 ? ». Si les réponses sont « une colonne statut, un cron, un export SQL, on perd le fil » — c'est exactement le périmètre où Camunda apporte le plus.

Chapitre 05 · Infra

L'infra : Docker & Kubernetes, expliqués simplement

Vocabulaire minimal : une image Docker = une boîte contenant l'appli et tout ce qu'il lui faut pour tourner. Kubernetes = le gestionnaire qui fait tourner ces boîtes (« pods »), les redémarre si elles tombent, et les multiplie si la charge monte. Un namespace = un étage du bâtiment (un par environnement).

5.1 — Conteneuriser un service Java

credit-orchestrator/Dockerfile — build en 2 étages
# Étage 1 : compiler (l'outillage lourd reste ici)
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -q        # cache des dépendances
COPY src ./src
RUN mvn package -DskipTests -q

# Étage 2 : exécuter (image finale minimale)
FROM eclipse-temurin:17-jre-alpine
RUN addgroup -S app && adduser -S app -G app
USER app                                 # jamais root en banque
COPY --from=build /app/target/*.jar /app.jar
EXPOSE 8080
ENTRYPOINT ["java","-XX:MaxRAMPercentage=75.0","-jar","/app.jar"]

Même modèle pour les 4 workers. Pour les remotes Angular : étage 1 node:20 qui fait npm ci && npm run build, étage 2 nginx:alpine qui sert les fichiers statiques (dont le fameux remoteEntry.js).

5.2 — La topologie Kubernetes

Fig. 6 — Un namespace par environnement. Les 2 replicas de l'orchestrateur partagent la même base : Camunda 7 est conçu pour ça.

5.3 — Les 3 points d'attention spécifiques à Camunda

  • La base n'est PAS jetable. Elle contient les dossiers en cours de dizaines de clients. PostgreSQL managé (ou géré par les DBA), sauvegardes testées, et les scripts SQL de migration Camunda appliqués lors des montées de version du moteur (jamais schema-update: true hors du poste de dev).
  • Le rolling update est naturellement sûr : pendant un déploiement, ancienne et nouvelle version de l'orchestrateur coexistent quelques secondes sur la même base — c'est supporté. Les instances en cours continuent sur leur version du BPMN (le versioning du chap. 13 du guide).
  • Un seul deployer de BPMN : les processus sont déployés par l'application au démarrage. Ne déployez jamais un BPMN « à la main » via l'API REST en prod — sinon Git ne reflète plus ce qui tourne.

Chapitre 06 · CI/CD

CI/CD avec GitHub Actions

Vocabulaire minimal : un workflow GitHub Actions = un fichier YAML dans .github/workflows/ qui décrit « quand il se passe X sur le repo, exécute ces étapes ». Un runner = la machine qui exécute (hébergée par GitHub, ou self-hosted dans le réseau de la banque — quasi obligatoire pour toucher le cluster). Un secret = une valeur chiffrée injectée au pipeline sans jamais apparaître dans le code.

6.1 — La philosophie : chaque commit prouve qu'il est sain

Fig. 7 — Le chemin d'un commit. La particularité Camunda : les tests de processus sont dans le pipeline — un gateway mal câblé bloque la PR, comme n'importe quel bug.

6.2 — Le workflow complet de l'orchestrateur

credit-orchestrator/.github/workflows/ci-cd.yml
name: CI/CD credit-orchestrator

on:
  pull_request:
  push:
    branches: [main]

env:
  IMAGE: ghcr.io/banque/credit-orchestrator

jobs:
  # ---------- 1. Construire et prouver ----------
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: '17', cache: maven }

      - name: Tests (unitaires + processus BPMN)
        run: mvn -B verify

      - name: Publier la couverture de processus (HTML visuel)
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: process-coverage
          path: target/process-test-coverage/

  # ---------- 2. Image Docker (uniquement sur main) ----------
  package:
    needs: build-test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions: { contents: read, packages: write }
    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build & push (tag = sha)
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ${{ env.IMAGE }}:${{ github.sha }}

      - name: Scan de vulnérabilités
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.IMAGE }}:${{ github.sha }}
          severity: HIGH,CRITICAL
          exit-code: '1'        # image vulnérable = pipeline rouge

  # ---------- 3. Déclencher le déploiement DEV (GitOps) ----------
  deploy-dev:
    needs: package
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Mettre à jour le dépôt credit-platform-config
        uses: actions/checkout@v4
        with:
          repository: banque/credit-platform-config
          token: ${{ secrets.PLATFORM_CONFIG_PAT }}
      - run: |
          cd envs/dev/credit-orchestrator
          yq -i '.spec.template.spec.containers[0].image =
                 "${{ env.IMAGE }}:${{ github.sha }}"' deployment.yaml
          git config user.name "ci-bot"
          git config user.email "ci@banque.ma"
          git commit -am "deploy(dev): credit-orchestrator ${{ github.sha }}"
          git push

Trois choses à bien voir dans ce YAML :

  • mvn verify exécute les tests de processus : le moteur H2 en mémoire déroule le BPMN. Si quelqu'un casse une condition de gateway, la PR est rouge avant la revue humaine.
  • Le tag de l'image est le SHA du commit : traçabilité absolue. « Quelle version tourne en prod ? » → un SHA → un commit → un diff. Jamais de tag latest en banque.
  • Le job de déploiement ne parle pas au cluster : il modifie un fichier YAML dans le dépôt GitOps. Le déploiement réel, c'est le chapitre suivant.

6.3 — Le workflow d'un micro-frontend

Même squelette, adapté : npm ci → lint + tests (Jest/Karma) → ng build du remote → image nginx → GHCR → MàJ GitOps. Une étape spécifique MF mérite d'exister :

mf-corbeille/.github/workflows/ci-cd.yml (l'étape qui évite les mauvaises surprises)
      - name: Vérifier le contrat Module Federation
        run: |
          # le build doit produire remoteEntry.js et exposer './routes'
          test -f dist/mf-corbeille/remoteEntry.js
          node scripts/check-exposed-modules.js   # compare aux attentes du shell
💡 L'autonomie MF, concrétisée par le pipeline

Karim signale un libellé trompeur dans la corbeille à 11h. Fix, PR, revue, merge : à 11h40 la nouvelle image nginx de mf-corbeille est en dev, en UAT à 14h, en prod le soir même. Le shell, l'orchestrateur, les workers : rien d'autre n'a bougé, personne d'autre n'a été sollicité. C'est tout l'intérêt de l'architecture : le rayon d'impact d'un changement = le repo qui change.

6.4 — Et le BPMN dans tout ça ?

Le .bpmn suit exactement le même chemin qu'une classe Java : PR → tests de processus → revue (le Camunda Modeler sait afficher le diff visuel entre deux versions du fichier) → merge → image → déploiement. Au démarrage du nouveau pod, le moteur détecte que le fichier a changé et crée la version N+1 de la définition. Les dossiers en cours terminent sur la version N ; les nouveaux démarrent sur la N+1. Un changement de processus métier est donc livré comme n'importe quel changement de code — même rigueur, même traçabilité, même rollback (redéployer l'image précédente).

Chapitre 07 · GitOps

GitOps : la promotion dev → UAT → prod

L'idée GitOps en une phrase : le cluster copie Git, jamais l'inverse. Le dépôt credit-platform-config décrit l'état voulu de chaque environnement ; un agent dans le cluster (Argo CD) compare en continu « voulu vs réel » et applique les écarts. Personne n'a besoin des droits kubectl sur la prod — il suffit de savoir faire une PR.

7.1 — Le dépôt GitOps

credit-platform-config/
envs/
├── dev/
│   ├── credit-orchestrator/deployment.yaml   image: ...:a1b2c3d  ← mis à jour par la CI
│   ├── credit-scoring-svc/...
│   └── mf-corbeille/...
├── uat/
│   └── (mêmes fichiers, SHAs plus anciens = versions validées)
└── prod/
    └── (mêmes fichiers, SHAs encore plus stables)

7.2 — La promotion = une Pull Request

Fig. 8 — Promouvoir = copier un SHA d'un dossier à l'autre, via PR. L'historique Git devient le registre d'audit des mises en production — précieux face à un auditeur.

Les garde-fous, tous natifs GitHub :

  • Branch protection sur main du dépôt GitOps : pas de push direct, revues obligatoires, CODEOWNERS (envs/prod/** @squad-credit-leads @ops-team).
  • GitHub Environments (uat, prod) avec required reviewers : le job de déploiement prod attend une approbation humaine dans l'interface GitHub — horodatée, nominative, auditable.
  • Rollback = git revert de la PR de promotion. Argo CD redéploie l'ancienne image en ~1 minute. (Rappel Camunda : les instances démarrées sur la version N+1 du BPMN doivent rester exécutables — gardez les delegates compatibles, ou traitez ces instances via Cockpit.)
⚠ La config sensible ne passe JAMAIS par Git

Le dépôt GitOps contient la structure (deployments, replicas, versions, ConfigMaps non sensibles). Les mots de passe base, secrets Keycloak, clés du prestataire de signature vivent dans Vault (ou en Sealed Secrets chiffrés). Un secret en clair dans un repo — même privé — est un incident de sécurité en banque, pas une négligence.

Chapitre 09 · Synthèse

Récap & glossaire

9.1 — Ce qu'il faut retenir, en 6 lignes

  1. Un domaine, une squad, huit repos : le BPMN, les services, les remotes ET le GitOps appartiennent à la même équipe.
  2. Le front ne connaît que la façade métier ; Camunda est un détail d'implémentation invisible depuis le navigateur.
  3. Les workers tirent le travail (External Tasks) : découplage, retries, idempotence — la panne d'un service met le travail en file, pas le système à genoux.
  4. Le BPMN est du code : testé dans la PR, versionné, livré par le même pipeline, rollbackable.
  5. GitHub Actions construit et prouve ; Git déploie (GitOps) : la prod = le contenu du dossier envs/prod, approuvé par PR.
  6. En incident, l'état est roi : Cockpit montre chaque dossier figé, le replay est un clic, l'idempotence évite les doublons.

9.2 — Glossaire minute

Orchestrateur

Le microservice qui embarque le moteur Camunda et « connaît la partition » : l'ordre des étapes du parcours. Il ne fait aucun travail métier lourd lui-même.

External Task / topic / worker

Une étape du processus publiée sur un « tableau d'offres » (le topic). Un worker est un service qui vient chercher ces offres, fait le travail, et rend le résultat. C'est le pont entre Camunda et vos microservices.

Business key

Votre référence métier (CR-2026-08812) attachée à l'instance. Elle relie tout : le front, les logs de tous les services, Cockpit, l'historique.

Micro-frontend / remote / shell

Le shell est le cadre du puzzle (menu, login) ; un remote est une pièce (la corbeille), construite et déployée par sa propre équipe, chargée dynamiquement par le shell via Module Federation.

GitOps / Argo CD

Pratique où un dépôt Git décrit ce qui doit tourner sur le cluster, et un agent applique en continu. Déployer = merger une PR ; auditer = lire l'historique Git ; rollback = revert.

Runner self-hosted

Une machine du réseau de la banque qui exécute les jobs GitHub Actions ayant besoin d'accéder aux ressources internes (registre privé, cluster) — les runners publics de GitHub n'y ont pas accès.

Incident (au sens Camunda)

Un dossier figé après épuisement des retries d'une étape technique. Visible dans Cockpit avec sa cause ; relançable en un clic une fois le problème réparé. Rien n'est perdu.

9.3 — Par où commencer sur la refonte BMCI ?

Le chemin le moins risqué, éprouvé dans ce type d'organisation : choisir un parcours à forte douleur (statuts éparpillés, relances manuelles, « où en est le dossier ? » quotidien), le modéliser en atelier avec le métier, monter le squelette technique de ce document (orchestrateur + 1 worker + 1 remote + pipeline), et livrer une première version fine mais complète — de la PR jusqu'à Cockpit en prod. La deuxième équipe qui voudra son parcours copiera vos repos comme templates : c'est comme ça qu'une plateforme naît.
