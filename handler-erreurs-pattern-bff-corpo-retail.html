2L'argumentaire pour Yassin

Le pitch en 3 phrases : « J'ai repris le pattern du BFF : même contrat ErrorDTO, même mécanique MessageSource + registre de codes, même scoping. J'ai adapté un seul point au contexte : nous sommes un microservice producteur d'erreurs, donc le code sort du type de l'exception plutôt que d'une regex sur le message — plus fiable, et nos messages restent préfixés [code] pour que TON handler BFF les repêche tel quel. J'ai aussi gardé les fieldErrors sur les 400 en étendant ErrorDTO sans le casser. »

Q&A anticipées

Question probableRéponse
« Pourquoi pas la regex comme chez nous ? »Elle résout un problème que le BFF a (retrouver un code dans le message d'un service aval) et que nous n'avons pas (nous créons l'exception, le type EST le code). En prime, notre préfixe [code] alimente précisément ta regex quand tu propages nos erreurs.
« Pourquoi plusieurs @ExceptionHandler au lieu d'un seul ? »Pour distinguer erreur métier attendue (4xx propre, pas de log d'erreur) et bug technique (500 + log.error avec stacktrace). Un catch-all unique déguise les bugs en erreurs fonctionnelles et les rend invisibles en supervision.
« FieldErrorsDTO, ce n'est pas une divergence de contrat ? »Non : héritage additif. code et message sont toujours là, au même endroit — un champ en plus sur un seul cas (400 de validation). Si tu la refuses, la retirer = 3 lignes.
« Et le 999 ? »Remplacé par un vrai 500 : 999 n'est pas un statut HTTP, certains clients/proxys le gèrent mal. Le fallback est CORPO_TECHNICAL_ERROR(500).
« Pourquoi ErrorDTO dans le module common ? »Pour que le retail réutilise le même fichier lors de sa migration (guide fourni au collègue) — un seul contrat d'erreur pour tout le microservice.

package com.bmci.workflow.fund.management.common.dto;

import java.util.Map;

/**
 * Extension ADDITIVE du contrat BFF pour les erreurs de validation de
 * surface (400) : ajoute le detail champ par champ SANS casser le contrat
 * — un client qui ne lit que {code, message} continue de fonctionner.
 */
public class FieldErrorsDTO extends ErrorDTO {

    private Map<String, String> fieldErrors;

    public FieldErrorsDTO() {
    }

    public FieldErrorsDTO(String code, String message, Map<String, String> fieldErrors) {
        super(code, message);
        this.fieldErrors = fieldErrors;
    }

    public Map<String, String> getFieldErrors() {
        return fieldErrors;
    }

    public void setFieldErrors(Map<String, String> fieldErrors) {
        this.fieldErrors = fieldErrors;
    }
}

/**
 * Registre des erreurs corpo — meme role que le CustomHttpStatus du BFF :

@Getter
@AllArgsConstructor
public enum CorpoHttpStatus {

    CORPO_BAD_REQUEST(400, HttpStatus.BAD_REQUEST, "corpo.error.badrequest"),
    CORPO_NOT_FOUND(404, HttpStatus.NOT_FOUND, "corpo.error.notfound"),
    CORPO_CONFLICT(409, HttpStatus.CONFLICT, "corpo.error.conflict"),
    CORPO_VALIDATION_ERROR(422, HttpStatus.UNPROCESSABLE_ENTITY, "corpo.error.validation"),
    CORPO_TECHNICAL_ERROR(500, HttpStatus.INTERNAL_SERVER_ERROR, "corpo.error.technical");

    private final int code;
    private final HttpStatus httpStatus;
    private final String messageKey;
}

package com.bmci.workflow.fund.management.application.rest.corpo.error;

import com.bmci.q360.domain.corpo.exceptions.CorpoConflictException;
import com.bmci.q360.domain.corpo.exceptions.CorpoNotFoundException;
import com.bmci.q360.domain.corpo.exceptions.CorpoValidationException;
import com.bmci.workflow.fund.management.application.rest.corpo.CorpoDemandController;
import com.bmci.workflow.fund.management.common.dto.ErrorDTO;
import com.bmci.workflow.fund.management.common.dto.FieldErrorsDTO;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.context.MessageSource;
import org.springframework.http.ResponseEntity;
import org.springframework.http.converter.HttpMessageNotReadableException;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

import java.util.HashMap;
import java.util.Locale;
import java.util.Map;

/**
 * Handler d'erreurs corpo ALIGNE sur le pattern du socle BFF orchestrator :
 * meme contrat de reponse (ErrorDTO {code, message}), meme mecanique
 * MessageSource + registre de statuts (CorpoHttpStatus ~ CustomHttpStatus),
 * meme scoping par assignableTypes.
 * Adaptation assumee au contexte "microservice producteur d'erreurs" :
 * resolution par TYPE d'exception (pas de regex sur les messages — le code
 * vient du type, source fiable), fieldErrors conserves sur les 400 de
 * validation (extension additive du contrat), et filet final Exception.class
 * avec log.error systematique (un bug reste visible, jamais silencieux).
 */
@Slf4j
@RestControllerAdvice(assignableTypes = CorpoDemandController.class)
@RequiredArgsConstructor
public class CorpoDemandExceptionHandler {

    private final MessageSource messageSource;

    // ==================== Erreurs metier (exceptions typees) ====================

    @ExceptionHandler(CorpoValidationException.class)
    public ResponseEntity<ErrorDTO> handleValidation(CorpoValidationException exception) {
        return build(CorpoHttpStatus.CORPO_VALIDATION_ERROR, exception.getMessage());
    }

    @ExceptionHandler(CorpoNotFoundException.class)
    public ResponseEntity<ErrorDTO> handleNotFound(CorpoNotFoundException exception) {
        return build(CorpoHttpStatus.CORPO_NOT_FOUND, exception.getMessage());
    }

    @ExceptionHandler(CorpoConflictException.class)
    public ResponseEntity<ErrorDTO> handleConflict(CorpoConflictException exception) {
        return build(CorpoHttpStatus.CORPO_CONFLICT, exception.getMessage());
    }

    // ==================== Erreurs de surface (contrat HTTP) ====================

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorDTO> handleBeanValidation(MethodArgumentNotValidException exception) {
        Map<String, String> fieldErrors = new HashMap<>();
        exception.getBindingResult().getFieldErrors().forEach(fieldError ->
                fieldErrors.put(fieldError.getField(), fieldError.getDefaultMessage()));

        CorpoHttpStatus status = CorpoHttpStatus.CORPO_BAD_REQUEST;
        String message = resolveMessage(status, "des champs sont invalides");
        return ResponseEntity.status(status.getHttpStatus())
                .body(new FieldErrorsDTO(String.valueOf(status.getCode()), message, fieldErrors));
    }

    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ResponseEntity<ErrorDTO> handleUnreadable(HttpMessageNotReadableException exception) {
        return build(CorpoHttpStatus.CORPO_BAD_REQUEST, "le corps de la requête est illisible ou mal formé");
    }

    // ==================== Filet final : jamais silencieux ====================

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorDTO> handleUnexpected(Exception exception) {
        log.error("Erreur non prévue sur un endpoint corpo", exception); // trace complete obligatoire
        return build(CorpoHttpStatus.CORPO_TECHNICAL_ERROR, null);
    }

    // ==================== Fabrique commune ====================

    private ResponseEntity<ErrorDTO> build(CorpoHttpStatus status, String detail) {
        return ResponseEntity.status(status.getHttpStatus())
                .body(new ErrorDTO(String.valueOf(status.getCode()), resolveMessage(status, detail)));
    }

    private String resolveMessage(CorpoHttpStatus status, String detail) {
        return messageSource.getMessage(status.getMessageKey(),
                new Object[]{detail == null ? "" : detail}, Locale.FRANCE);
    }
}

4Étapes d'exécution & impacts

  1. Créer ErrorDTO et FieldErrorsDTO dans le module common (le pom du common n'a besoin de rien de plus).
  2. Créer CorpoHttpStatus et messages.properties.
  3. Remplacer CorpoDemandExceptionHandler. Supprimer l'import ProblemDetail devenu inutile — rien d'autre ne bouge (c'est le bénéfice d'avoir isolé le contrat d'erreur dans l'advice).
  4. Adapter le CorpoDemandControllerTest (extrait ci-dessus) → mvn clean install vert.
  5. Rejouer les cas d'erreur du cahier de tests (ER-01 → ER-13) : les statuts HTTP ne changent pas, seul le corps change — {"code":"422","message":"[422] La coupure…"} au lieu du ProblemDetail, et fieldErrors toujours présent sur les 400. Bonus nouveau : le cas « même coupure en double » (uq_corpo_od) sort maintenant en 500 propre {code:"500", message:"[500] Une erreur technique…"} avec stacktrace dans les logs — plus de page d'erreur brute.
Impact à signaler (rien en silencieux) : le format du corps d'erreur change (ProblemDetail → ErrorDTO). Si le front a déjà branché les erreurs corpo, le prévenir : detail devient message, status devient code (chaîne), les codes CORPO_* textuels deviennent numériques ("400", "404"…) — la granularité métier reste portée par le libellé. C'est LE format cible validé par la note du tech lead, autant basculer avant que le front ne branche.

5Guide de réplication retail (à transmettre à ton collègue)

Le gros du travail est déjà mutualisé : ErrorDTO et FieldErrorsDTO sont dans le common, il les importe tel quel. Il lui reste 4 étapes, sans big-bang :

  1. Son registre : créer RetailHttpStatus dans rest.retail.error — copier CorpoHttpStatus, renommer les constantes (RETAIL_BAD_REQUEST 400, RETAIL_NOT_FOUND 404, RETAIL_CONFLICT 409, RETAIL_VALIDATION_ERROR 422, RETAIL_TECHNICAL_ERROR 500) et les clés (retail.error.*).
  2. Ses libellés : ajouter les clés retail.error.* dans le MÊME messages.properties (fichier partagé du container).
  3. Son advice : créer RetailDemandExceptionHandler dans rest.retail.error, scopé assignableTypes = CashDemandController.class — même squelette que le corpo. Particularité retail : pas encore d'exceptions typées, tout sort en SessionServiceException → dans un premier temps, un handler SessionServiceException → 500 (comportement actuel préservé, mais au format ErrorDTO) + le filet Exception.class avec log.error. Zéro régression fonctionnelle, contrat unifié immédiatement.
  4. Phase 2 (progressive, à son rythme) : créer ses exceptions typées (RetailNotFoundException…) dans domain.retail, remplacer les RuntimeException("Cash demand not found") de l'adapter par les typées, retirer les try/catch des use cases méthode par méthode — chaque méthode migrée gagne son vrai statut (404/409/422) au lieu du 500. Le corpo lui sert de modèle ligne à ligne.
Résultat final pour le microservice : un seul contrat d'erreur (celui du BFF), deux registres par périmètre, un fichier de messages, deux advices scopés — et une migration retail sans big-bang. C'est l'application complète de la note 5, défendable en une démo de 5 minutes.
