00 Orientation

The brief, and why 2026 is the perfect moment for this

BMCI wants to break a set of services into micro-frontends, built with Webpack, possibly mixing Angular, React and Vue. That's a microservices philosophy applied to the UI — and you already think in those terms from your backend work.

Here's the single most useful thing to understand before anything else: the whole frontend world converged on the same idea in the last 18 months. Angular went signal-first and zoneless, Vue shipped Vapor Mode (no virtual DOM), React shipped a compiler that auto-optimizes. Different roads, one destination: the framework should know exactly what changed and touch only that. When you talk to the BMCI team, framing things around this shared direction will make you sound like someone who sees the forest, not just the trees.

01 Foundations

What a micro-frontend actually is

A micro-frontend architecture splits a big single-page app into independently developed, built and deployed UI pieces, assembled into one experience the user sees as a single app.

The vocabulary (only three words)

  • Shell / Host — the container application. It owns the shared layout, top-level routing, authentication, and the common libraries. It loads the pieces.
  • Remote — a separately built and deployed application (a "vertical" of the product) that exposes parts of itself (a component, a route, a feature) to be loaded by the host at runtime.
  • Shared dependencies — libraries (Angular, React, a design system) that the host and remotes agree to load once and reuse, instead of each shipping its own copy.

The senior take — when NOT to do this Micro-frontends are organizational medicine, not a performance trick. They pay off when you have multiple teams that need to ship independently. For a single small team, they add real cost: distributed-systems problems land in the browser (remote availability, version drift, network errors mid-load), shared dependencies get tricky, and CSS/state can leak across boundaries. If BMCI has the team structure to justify it, great — but be ready to say "a well-modularized monolith is the right answer until team independence is the actual bottleneck." That sentence shows judgment.

02 The techniques

How you actually stitch remotes into a shell

There are several ways to compose micro-frontends. The first split to understand is build-time vs run-time composition.

Build-time composition

Remotes are pulled in as packages and bundled together at build. Simple, type-safe — but you lose independent deployment, because changing one piece means rebuilding the host. Rarely "true" micro-frontends.

Run-time composition ✓

The host loads remotes at runtime from URLs. A remote can be redeployed and the host picks up the new version without rebuilding. This is what people mean by micro-frontends, and what Module Federation does.

The five run-time approaches

ApproachHow it worksBest for
Module Federation 2.0Webpack/Rspack/Vite plugin: remotes expose modules via a remoteEntry.js manifest; host imports them dynamically and shares singletons.The default in 2026 when teams share a framework. Best DX, type sharing, dependency sharing.
Native FederationSame mental model, but browser-native using ES Modules + Import Maps; bundler-agnostic. The Angular path.Modern Angular (esbuild builder), and anyone wanting standards over a webpack-specific runtime.
Web ComponentsEach remote is wrapped as a Custom Element (<cards-widget>); the shell just places the tag. Framework-neutral contract.Genuinely mixed frameworks (Angular + React + Vue) living together.
single-spaAn orchestrator with lifecycle hooks (bootstrap/mount/unmount) per app; framework-agnostic router.Legacy + modern coexisting (AngularJS, jQuery) or hard multi-framework cases.

Clarify the "Webpack" assumption on day one When the interviewer said "Webpack," it likely means Module Federation (which began as a Webpack 5 feature). Knowing this distinction on arrival is a strong first impression.
03 The core engine

Module Federation 2.0, in depth

This is the heart of the refonte. Master the host/remote model, shared singletons, dynamic loading, and resilience, and you can hold any architecture conversation.

The mechanics

A remote builds a special entry file — remoteEntry.js — that lists what it exposes and which dependencies it shares. The host declares which remotes it consumes and which dependencies it shares. At runtime the host fetches the remote's manifest, negotiates shared versions, and dynamically imports the exposed module.

sequenceDiagram
  participant B as Browser
  participant H as Shell Host
  participant R as Payments Remote
  B->>H: load shell bundle
  Note over H: user navigates to /payments
  H->>R: fetch remoteEntry.js manifest
  R-->>H: exposed modules and shared versions
  Note over H,R: negotiate singletons — load framework once
  H->>R: import payments/Checkout
  R-->>B: Checkout renders into the shell slot
Runtime federation: the host discovers and loads a remote on demand, sharing common libraries.

The remote — exposing a feature

// webpack.config.js — the Payments remote (React example)
new ModuleFederationPlugin({
  name: 'payments',
  filename: 'remoteEntry.js',            // the manifest the host fetches
  exposes: {
    './Checkout': './src/Checkout',    // what the world can import
  },
  shared: {
    react:       { singleton: true, requiredVersion: '^19.0.0' },
    'react-dom': { singleton: true, requiredVersion: '^19.0.0' },
  },
})

The host — consuming it

// webpack.config.js — the Shell / Host
new ModuleFederationPlugin({
  name: 'shell',
  remotes: {
    payments: 'payments@https://payments.bmci.ma/remoteEntry.js',
    cards:    'cards@https://cards.bmci.ma/remoteEntry.js',
  },
  shared: { react: { singleton: true }, 'react-dom': { singleton: true } },
})

// in the host app — lazy load + Suspense + an error boundary
const Checkout = lazy(() => import('payments/Checkout'));
<ErrorBoundary fallback={<CheckoutUnavailable/>}>
  <Suspense fallback={"Loading…"}><Checkout/></Suspense>
</ErrorBoundary>

The four ideas that actually matter

ConceptWhy you must get it right
singleton: trueForces one shared instance of a library. Mandatory for React, Angular, Vue and any global state — two copies of React in one page break hooks and context. This is the #1 source of MFE bugs.
requiredVersion / strictVersionVersion negotiation. If a remote needs a version the host can't satisfy, you decide whether to fail loudly (strictVersion) or fall back. Version drift across teams is the hard operational problem.
Dynamic remotesInstead of hard-coding remote URLs at build, read them from a manifest/config at runtime — so you can add or move a remote without rebuilding the shell. Essential for independent deployment.
Error boundaries & fallbacksA remote is a network call that can fail. Always wrap it so one dead remote degrades to a fallback instead of crashing the shell.
What 2.0 added (and why it matters for a bank)
  • Decoupled runtime — federation logic is separate from the bundler, so the same model works on webpack, Rspack and Vite. Migration is no longer "rip out your toolchain."
  • Automatic TypeScript types across remotes — importing payments/Checkout is now type-checked. This kills a whole class of integration bugs — big for a regulated codebase.
  • Manifest-based discovery, preloading, runtime plugins, Chrome DevTools — better operability and a fix for the "request waterfall" (loading remotes serially) via preloading.
  • Node.js runtime support — remotes can be consumed by SSR and BFF layers, unifying delivery front-to-back.
The "request waterfall" — name it before they do A known MFE problem: the shell loads, then fetches a remote manifest, then fetches its chunks, then its shared deps — each step waiting on the last, hurting first paint. 2.0's answer is preloading and SSR/data prefetch. Mentioning the waterfall and its mitigation signals you've thought past the happy path.

The performance reality (so you can reassure them)

Done right, federation is cheap: the shell loads instantly, remotes lazy-load on route entry, the shared framework loads once, and each remote keeps its own cache fingerprint so deploying Cards doesn't invalidate the Accounts cache. A typical three-remote setup adds only tens of milliseconds to first meaningful paint versus a monolith — a fair price for the deployment independence.

Backend parallel to lean on The host/remote split is your gateway/microservice split. singleton sharing is the frontend version of "don't run five copies of the same library." Error boundaries are your circuit breakers. Dynamic remote config is service discovery for the UI. When you explain MFE to colleagues, use these analogies — they land instantly with backend people.

Be ready to answer

  • Host vs remote — what does each declare?
  • What is remoteEntry.js?
  • Why must shared framework deps be singletons?
  • How do you handle a remote that fails to load?
  • What changed in Module Federation 2.0?
  • What is the request-waterfall problem and how is it mitigated?
04 The hard part · read this twice

Mixing Angular, React and Vue in one app

This is the question where most people hand-wave. If you can speak to it precisely, you'll be the most credible person in the room.

The honest truth Module Federation shines when everyone shares one framework — then Angular/React loads once and is shared. The moment you genuinely mix Angular + React + Vue, you cannot share a framework runtime across them: the page ships Angular and React and Vue. That's heavier bundles, more memory, and a harder integration story. So multi-framework is a capability to use deliberately — usually to absorb a legacy app during migration — not a default to reach for.

The integration contract: Web Components

When frameworks must coexist, the clean boundary is the browser's own standard: Custom Elements. Each remote wraps its root as a Web Component (Angular Elements, React via a wrapper, Vue's defineCustomElement). The shell just places a tag and passes data via attributes/properties; the remote emits custom events outward. Neither side needs to know the other's framework.

The four cross-cutting problems (and the patterns that solve them)

ProblemPattern
Communication between remotesKeep it loose: a shared event bus / browser CustomEvents, or a tiny framework-agnostic shared store. Avoid one framework's state library leaking across boundaries.
Shared state (e.g. logged-in user)Put it in the shell and pass down via props/attributes or a shared singleton service; treat cross-remote state as you would shared data between microservices — minimal and explicit.
Styling collisionsScope everything: Shadow DOM (with Web Components), CSS modules, or a strict design-system token layer. CSS leakage is the classic MFE bug.
RoutingThe shell owns top-level routing and delegates a URL sub-tree to each remote, which runs its own internal router. Agree the boundary early.
Say it like this — the multi-framework answer "Module Federation is ideal when we standardize on one framework, because then it loads once and we share it. If we truly need Angular, React and Vue together — usually to migrate a legacy app gradually — I'd use Web Components as the neutral contract: each remote exposes a custom element, the shell passes data via attributes and listens to custom events, and we keep cross-remote state minimal and explicit. But I'd push to converge on one framework for new remotes, because multi-framework means shipping multiple runtimes and paying for it in bundle size and memory."
Where you're strong / where to study Your Angular depth (Natixis, Angular 21/22) makes you the natural owner of the shell and the Angular remotes. Your past React work (BNP) means you're not starting cold on the React side. Vue is the one to put real hours into before day one — the good news is Vue's Composition API will feel familiar coming from Angular's reactive model.
05 Framework backbone · Angular

Angular — the backbone (your home turf)

Angular is the batteries-included, TypeScript-first, opinionated framework: routing, forms, HTTP, DI and testing all ship in the box. In 2026 it's in its "signal-first" era.

The pillars

  • Components & standalone — UI is a tree of components. Since v19 standalone components are the default (no more NgModules boilerplate).
  • Dependency Injection — Angular's hierarchical DI is its superpower and matches your backend Spring instincts: declare what you need, the framework provides it.
  • Templates & control flow — declarative HTML with the built-in @if / @for / @switch control flow (since v17).
  • Signals — the reactive primitive. A signal is a value that tracks who reads it and notifies them precisely when it changes.
  • RxJS — still there for streams and complex async; signals reduce how often you reach for it, but they coexist.
// Angular 22 — signals + OnPush-by-default + standalone
import { Component, signal, computed } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `{{ count() }} → doubled {{ doubled() }}`,
})
export class Counter {
  count   = signal(0);                       // writable signal
  doubled = computed(() => this.count() * 2); // derived, cached
  inc() { this.count.update(v => v + 1); }
}
Angular 22 — what's current (June 2026)
  • Signal-first & zoneless — new apps no longer ship Zone.js; signals drive change detection, giving smaller bundles and more predictable updates.
  • OnPush is the default change detection strategy now (the old "check everything" is renamed Eager).
  • Signal Forms and the Resource API / httpResource are stable — a signal-based way to handle forms and async data with less boilerplate.
  • Vitest replaced Karma for testing; Angular Aria ships accessible UI primitives (relevant to your WCAG work).
  • Build is esbuild-based (fast); @boundary template error boundaries are in preview.

Angular + micro-frontends = Native Federation

Because Angular dropped Webpack for esbuild, classic Module Federation doesn't fit the default builder. The community answer — endorsed on the Angular blog — is Native Federation (@angular-architects/native-federation): the same host/remote/shared mental model and API, but implemented with browser-native ES Modules + Import Maps, delegating to Angular's esbuild builder. It supports SSR and incremental hydration, and can even interoperate with Webpack-based Module Federation remotes during a migration.

// federation.config.ts — an Angular shell with Native Federation
import { withNativeFederation, shareAll } from '@angular-architects/native-federation/config';

export default withNativeFederation({
  name: 'shell',
  remotes: { dashboard: 'http://localhost:4202/' },
  shared: { ...shareAll({ singleton: true, strictVersion: true }) },
});

// route that loads a remote on demand
{ path: 'dashboard',
  loadComponent: () => loadRemoteModule('dashboard', './Component')
                        .then(m => m.DashboardComponent) }
The crisp distinction to remember Module Federation = Webpack-born runtime (now also Rspack/Vite). Native Federation = same idea, browser-standard (ESM + Import Maps), built for Angular's esbuild builder. Same mental model, different engine. For new Angular work in 2026, Native Federation is the future-proof default; teams report build times dropping from ~45s to under 10s versus Webpack MF.

06 Framework backbone · React

React — the backbone (your second language)

React is a library, not a full framework: it owns the view layer and leaves routing, data and structure to you (or to a meta-framework like Next.js). It's the most widely deployed of the three.

The pillars

  • Components & JSX — functions that return JSX (HTML-in-JS). Composition over inheritance.
  • One-way data flow — data flows down via props; changes flow up via callbacks. Predictable.
  • Hooks — useState, useEffect, useMemo, useContext… functions that add state and lifecycle to function components.
  • Virtual DOM & reconciliation — React builds a virtual tree, diffs it against the previous one, and patches only what changed. (This is the cost the React Compiler now minimizes.)
// React 19 — a function component with a hook
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
React 19 / 19.2 — what's current (2026)
  • React Compiler (stable) — automatically memoizes; you mostly stop hand-writing useMemo/useCallback/memo. Big mental-load reduction.
  • Server Components (RSC) — components that render on the server, shipping less JavaScript to the browser. Powerful, but adds a server/client boundary to reason about.
  • Actions & Server Actions, useFormStatus, useOptimistic — built-in async form handling.
  • use hook (read a promise/context in render), ref as a prop (no more forwardRef), native document metadata, <Activity> (preserve state while hidden).
Banking-relevant security note A critical remote-code-execution vulnerability in React Server Components (CVE-2025-55182) was disclosed in late 2025 and patched in React 19.0.1 / 19.1.2 / 19.2.1. Pure client-side React apps weren't affected, but in a bank you'll want RSC-using services pinned to patched versions. Knowing this shows security awareness — exactly the BMCI instinct.

React in micro-frontends

React is the most common framework in Module Federation examples, so the path is well-trodden: expose components from a remote, share react and react-dom as singletons, wrap remote loads in an Error Boundary + Suspense. One caveat: Server Components don't compose cleanly across federated boundaries yet — for MFE remotes, prefer classic client components unless you have a deliberate SSR strategy.

07 Framework backbone · Vue

Vue — the backbone (the one to study)

Vue is the progressive, approachable framework: gentle to start, scales up with official routing and state. Its signature is the Single-File Component (template + script + style in one .vue file).

The pillars

  • Single-File Components (SFC) — <template>, <script setup>, <style scoped> together. The scoped styles solve CSS leakage out of the box — handy for MFEs.
  • Reactivity — ref() for values, reactive() for objects, computed() for derived state, watch() for side effects. This is conceptually very close to Angular signals, which is your shortcut into Vue.
  • Composition API — the modern way to organize logic (<script setup>), versus the older Options API. The Composition API will feel natural to you.
  • Template directives — v-if, v-for, v-model (two-way binding), @click.
// Vue 3.6 SFC — Composition API with <script setup>
<script setup>
import { ref, computed } from 'vue';
const count = ref(0);
const doubled = computed(() => count.value * 2);
</script>

<template>
  <button @click="count++">{{ count }} → {{ doubled }}</button>
</template>
Vue 3.6 — what's current (2026)
  • Vapor Mode — a compile target that skips the virtual DOM entirely, compiling templates to direct DOM operations. Opt-in per component with <script setup vapor>; performance lands near Solid.js / Svelte, with 20–50% smaller bundles for Vapor components.
  • Alien Signals — a faster reactivity engine under the hood.
  • Fully backward compatible — classic virtual-DOM Vue keeps working; you mix Vapor and VDOM via vaporInteropPlugin.
  • Caveats: Vapor needs <script setup> (no Options API yet), excludes <Suspense>, and some libraries are still catching up.
Your fast path into Vue Map it to what you know: Vue's ref/computed/watch ≈ Angular's signal/computed/effect; <script setup> ≈ a standalone component class; scoped styles ≈ Angular's view encapsulation; v-model ≈ two-way binding. You're not learning reactivity from zero — you're learning new syntax for a model you already understand. A weekend on the Composition API plus building one small SFC app will get you conversational.

Pick Angular when…

You need structure, large teams, long-lived enterprise apps, strong typing and conventions. The shell and core remotes are a natural fit.

Pick React when…

You want the biggest ecosystem and hiring pool, server-rendering options, and maximum flexibility. Most MFE tutorials assume it.

Pick Vue when…

You want fast onboarding, excellent performance (Vapor) for data-dense dashboards, and minimal ceremony.

Say it like this — when asked "which is best?" "None is 'best' in the abstract — they've converged on the same goals. For BMCI I'd weigh team skills, hiring, and the existing codebase over benchmarks. The architectural decision that matters more than the framework is: do we standardize on one for new remotes — which lets Module Federation share the runtime — or do we genuinely need several, in which case Web Components become our contract and we accept the bundle cost."
09 Production reality

Deployment, performance & security

This is where your banking instincts pay off — micro-frontends turn the frontend into a distributed system, and you already know how to run those responsibly.

Independent deployment

Each remote ships on its own pipeline to a CDN; the shell discovers it at runtime via a manifest.
  • Each remote = its own repo, CI pipeline, version and CDN path — exactly like a microservice.
  • Use a runtime manifest of remote URLs so adding/moving a remote needs no shell rebuild.
  • Per-remote cache fingerprints mean deploying one remote doesn't bust the others' caches.
  • Blue-green / canary applies per remote — flip the manifest to roll forward or back instantly.

Performance levers

  • Share singletons (framework, design system) so they load once.
  • Lazy-load remotes on route entry; preload likely-next remotes to beat the request waterfall.
  • Keep remotes small; watch total payload when multiple frameworks coexist.
  • Per-framework: Angular OnPush + signals; React Compiler; Vue Vapor for data-dense screens.

Security (the bank lens)

ConcernWhat to do
Loading code at runtimeOnly load remotes from trusted, HTTPS, same-origin-controlled hosts; lock down with a strict Content-Security-Policy; consider Subresource Integrity where applicable.
Auth across remotesCentralize OIDC in the shell (or a BFF). Remotes receive identity, never each re-implement login. (Your OIDC/BFF knowledge applies directly.)
Dependency / supply chainPin and scan dependencies; the React RSC CVE is a live reminder — keep federated remotes on patched versions.
IsolationShadow DOM / scoped styles to prevent one remote from interfering with another's DOM or CSS.
ResilienceError boundaries + fallbacks per remote, so a failed or compromised remote degrades gracefully.

Testing

Test each remote in isolation (unit + component: Vitest for Angular/Vue, Jest/Vitest + Testing Library for React), then add contract tests on the host↔remote interface (the exposed module's shape — now type-checked thanks to MF2) and a thin layer of end-to-end tests on the composed app for the critical journeys.

Your whole backend toolkit maps over Independent deploy = microservice CI/CD. Manifest of remotes = service discovery. Singletons = shared-library hygiene. Error boundaries = circuit breakers. CSP + dependency scanning = the supply-chain rigor a bank already demands. You're not learning operations from scratch — you're applying the operations you already do, to the UI.
