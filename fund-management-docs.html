Fund Management Microservice — the branch cash-supply workflow, end to end

A Spring Boot 3 / Java 17 hexagonal microservice that lets a bank branch create, update and cancel cash requests — supply or collection of physical money (banknotes and coins, per currency and per denomination) for ATMs and cash desks. This document explains what it does, how it's built, how to run and test it, and exactly where it needs to be fixed and hardened.

Java 17 Spring Boot 3.3.1 PostgreSQL · JPA/Hibernate Port 8300 Architecture Hexagonal (Ports & Adapters) Modules 5
01 · Functional

Functional intent — what problem this service solves

Every bank branch handles physical cash. ATMs run out of banknotes, cash desks accumulate more money than they are allowed to keep, and foreign-currency tills need restocking. Branches therefore need a formal, auditable workflow to ask the central cash service for money (a supply, French: alimentation) or to send excess money back (a collection, French: délestage / ramassage).

The Fund Management microservice digitizes that workflow. Its core business object is the Cash Request (CashRequest) — one demand emitted by a branch, on behalf of an identified user, for a given establishment. A request groups one or more Cash Operations, and each operation is broken down precisely into which currency, which denominations (200 MAD note, 100 MAD note, 10 MAD coin…), and how many of each.

The functional hierarchy

CashRequest ─ "Branch 012 of establishment BMCI asks for cash" (status: INITIATED → VALIDATED → CANCELLED)
 │
 ├── CashOperation #1 ─ SUPPLY an ATM, urgent, status INITIATED
 │     ├── OperationCurrencyDetail (MAD) — total 250 000,00
 │     │     ├── OperationDenomination: 200 MAD banknote × 1 000
 │     │     └── OperationDenomination: 100 MAD banknote ×   500
 │     └── OperationCurrencyDetail (EUR) — total 10 000,00
 │           └── OperationDenomination: 50 EUR banknote × 200
 │
 └── CashOperation #2 ─ COLLECTION from cash desk 001 → target desk 002
       └── ...
One request → N operations → N currency details per operation → N denomination lines per currency.

Business concepts & vocabulary

Business rules currently implemented

  • A request must contain at least one operation (CashRequest.validateRequest() — written but currently commented out in the create flow).
  • A cancelled request is frozen: no operation can be added or updated on it (addOperation / updateOperation guards), and the use case rejects update() on a CANCELLED request.
  • Cancelling a request cancels all of its operations (cascade of status in cancelById).
  • Operations can be cancelled individually; when the last active operation of a request is cancelled, the whole request auto-flips to CANCELLED (cancelOperations).
  • An already-cancelled operation cannot be cancelled twice (throws).
  • Denomination totals should be validated for supply-to-cash-desk operations: quantity × face value summed per line must produce the total (CashOperation.validateAndCalculateTotal — partially implemented, currently buggy, see §10).
  • Referential integrity by lookup: currencies are resolved by ISO code, denominations by label, establishments by code, users by uid — unknown values raise an error instead of silently creating garbage.
Where this fits in the bigger picture The parent groupId is com.bmci.workflow.system and the domain package is com.bmci.q360. This service is one microservice inside a broader BMCI branch-workflow platform ("Q360" / "qlick_flow" database). Left-over enums (IdentificationMethod, RegistrationReason, QlickeerEventName, IdentityMessage) and value objects (Gender, DocumentType…) were copied from a sibling identity/queue-management service and are dead code here — flagged in §10. Statuses like INITIATED → VALIDATED → CANCELLED strongly suggest a validation workflow (branch manager / cash center approval) that will later be orchestrated — a natural fit for Camunda 7, see §12.

Architecture — hexagonal, ports & adapters

The project follows the classic Clean / Hexagonal Architecture layout popularized for Spring microservices (domain-core → application-service → adapters → container). The rule that everything obeys: dependencies point inward. The domain core knows nothing about Spring, JPA or REST; adapters implement the interfaces (ports) the domain declares.

04 · Technical

Domain model in detail

DDD building blocks

  • Typed identifiers — every entity id is a value object wrapping a UUID (BaseId<T> → CashRequestId, CashOperationId, CurrencyId, DenominationId, OperationCurrencyDetailId, OperationDenominationId). This prevents accidentally passing an operation id where a request id is expected — a genuinely good practice.
  • Aggregate root — CashRequest extends AggregateRoot<CashRequestId>. It is the only entry point for mutating the cluster: addOperation(), updateOperation(), validateRequest() all guard against modifying a CANCELLED request.
  • Entities — CashOperation, OperationCurrencyDetail, Denomination extend the identity-equality BaseEntity<T> (equals/hashCode on id only).
  • Enums — RequestType { SUPPLY, COLLECTION } is used as the operation direction (operationSens); OperationType { ATM, CASH_DESK, CURRENCY } is the target.

Behavior living in the domain (as it should)

// CashRequest — invariant guards
public void addOperation(CashOperation operation) {
    if ("CANCELLED".equals(this.requestStatus))
        throw new IllegalStateException("cannot add an operation to a cancelled request");
    this.operations.add(operation);
}

// CashOperation — ventilation check for SUPPLY + CASH_DESK
public void validateAndCalculateTotal(CashOperation op, Map<String, BigDecimal> faceValuesMap) {
    // requires currencyDetails to be present, then sums faceValue × quantity per line
}
Bug inside validateAndCalculateTotal The lookup is faceValuesMap.get(opDenom.getDenomination()) — a Denomination object used as key in a Map<String, BigDecimal> keyed by label. It always returns null, so every valid ventilation throws "cupure inconu". It must be faceValuesMap.get(opDenom.getDenomination().getDenominationLabel()). Also the computed total is never stored (the totalAmount assignment is commented out), and the method takes itself as a parameter (op) while also being an instance method — it should just use this.

Status model (implicit state machine)

 CashRequest.requestStatus            CashOperation.operationStatus
 ─────────────────────────            ─────────────────────────────
   INITIATED ──▶ VALIDATED              INITIATED ──▶ (VALIDATED?)
       │                                    │
       └───────▶ CANCELLED                  └───────▶ CANCELLED
   (auto when all ops CANCELLED)        (individually, or cascaded from request)
Statuses are plain Strings today ("INITIATED", "VALIDATED", "CANCELLED") — a typed enum + explicit transition rules is a quick win (§12).

Data model (PostgreSQL)

Database qlick_flow, schema managed by Hibernate (ddl-auto: update). The tables as actually produced by the JPA entities (which are the source of truth today):

Schema drift: the SQL script no longer matches the entities The hand-written DDL script in the repo describes an older model: cash_request with unique_request_ref / requesting_branch / requester_ref / request_type, cash_operation with a direct currency_id (1:1 currency per operation), and operation_denomination as a pure N:N pivot with a composite PK. The entities have since evolved (currency moved to the intermediate operation_currency level, establishment & user extracted to referential tables, ids everywhere via BaseEntity). Because ddl-auto: update never drops anything, a database initialized from the script and then "updated" by Hibernate will contain a mix of live and dead columns, plus NOT NULL orphans that can break inserts. Action: retire the script, adopt Flyway/Liquibase as the single source of truth (§12).

Loading strategy

  • All @ManyToOne / @OneToOne are LAZY; open-in-view is false (good, forces conscious fetching).
  • CashDemandJpaRepository.findByIdWithOperation uses a join fetch on operations — but not on currency details / denominations / establishment / user, so mapping those outside a transaction throws LazyInitializationException (this is why getCashDemandById currently maps with the shallow legacy mapper).
  • Save/update flows preload the whole denomination and currency referentials into maps to resolve labels/ISO codes — fine for small referentials, replace by targeted findByLabelIn(...) queries if they grow.

07 · Technical

Request lifecycle — what happens on a create

POST /cash (JSON)
  └▶ CashDemandController.create(dto)
       └▶ CashDemandUseCaseImpl.create(dto)                       [@Transactional]
            1. CashRequestWebMapper.toDomain(dto)                  DTO → domain tree
            2. DenominationRepository.findAllActiveDenominations() label → faceValue map
            3. each operation.validateAndCalculateTotal(...)       business validation
            4. CashDemandRepository.save(domain)
                 └▶ CashDemandRepositoryImpl.save                  [@Transactional]
                      a. preload denominations (all) + currencies (map by ISO)
                      b. EstablishmentJpaRepository.findByEstablishmentCode(code)
                      c. UserProfileJpaRepository.findByUid(uid)
                      d. CashRequestDataAccessMappers.toPersistence(...)  domain → entity graph
                         (wires bidirectional parents, resolves FK targets, fails fast
                          on unknown currency / denomination)
                      e. cashDemandJpaRepository.save(entity)      cascade ALL persists the tree
                      f. toDomain(saved)                           entity → domain (full depth)
            5. CashRequestWebMapper.toResponseDTO(domain)          domain → DTO
  ◀ 200 OK + persisted aggregate

Cancel flows are simpler: load entity by id → mutate statuses in memory with the business checks → save → map back. Update is the destructive variant: existingEntity.getOperations().clear() then re-add all incoming operations (relying on orphanRemoval = true to delete the replaced rows).

3 · Build & run

# from the repo root
mvn clean install -DskipTests          # installs all 5 modules into the local repo

# run (only the container module is executable)
cd fund-management-container
mvn spring-boot:run
# or
java -jar target/fund-management-container.jar

The service listens on http://localhost:8300. show-sql: true prints every statement — useful now, disable in production.

Known build friction The root POM references ${maven-compiler-plugin.version} in the compiler plugin without defining that property — depending on your Maven setup this warns or fails; pin it (e.g. 3.13.0). 10 · Improve

Current issues — the audit

Ranked by severity. Blockers break requests today; majors corrupt behavior or data quality; minors are hygiene.

IDSeverityIssue & fix
I-01BlockergetAll() declares @PathVariable UUID id on GET /cash, which has no path variable → every call fails (MissingPathVariableException → 500). Fix: remove the parameter. Also decide between GET /cash/{id} and GET /cash/getCashDemandById — keep the RESTful one, delete the other.
I-02BlockervalidateAndCalculateTotal map lookup uses the wrong key (Denomination object vs String label) → every SUPPLY/CASH_DESK create with ventilation throws. Fix key, store the computed total into the operation, compare it to the declared totalAmountByCurrency, and make the method use this instead of receiving itself.
I-03BlockerSchema drift + ddl-auto: update (see §5). NOT NULL legacy columns (e.g. unique_request_ref) left in an old DB will make inserts fail even though the code is correct. Fix: Flyway baseline migration generated from the current entities, ddl-auto: validate.
I-04MajorRead endpoints use the legacy shallow mapper (CashRequestManualMapper): GET responses omit currency details, user, establishment on some paths — and getAll() maps lazy operations outside a transaction with open-in-view=false → LazyInitializationException risk. Fix: route all reads through mapperTemp1 generation, add fetch-joins (or projections) for list/detail, delete the legacy mappers.
I-05MajorNo error contract: every business failure is wrapped in SessionServiceException (misnamed — copied from another service) or raw RuntimeException → HTTP 500 for "not found", "already cancelled", "cannot update cancelled". Fix: dedicated exceptions (CashRequestNotFoundException, OperationAlreadyCancelledException…) + a @RestControllerAdvice mapping them to 404 / 409 / 422 with a ProblemDetail body (Spring 6 has it built in).
I-06MajorHardcoded credentials + internal IP in application.yml committed to the repo. Fix: environment placeholders (shown in §8), Spring profiles (local/dev/prod), secrets from vault/env. Also disable show-sql outside dev.
I-07MajorNo input validation: spring-boot-starter-validation is on the classpath but no @Valid, no constraints on DTOs. A create with null establishment/user reaches the DB and dies on a FK. Also update() does cashRequestDTO.getRequestStatus().toString() → NPE when status is absent. Fix: @NotBlank/@NotEmpty/@Valid cascade on the DTO tree + null-safe checks.
I-08MajorDomain purity broken: domain-core classes import jakarta.persistence (CashRequest, Currency, Denomination…) even though the module doesn't declare the dependency — it compiles only because it leaks transitively. Fix: strip JPA imports from domain-core; add the maven-enforcer plugin (banned dependencies) to keep it clean.
I-09Majorupdate() replace strategy is risky: clearing the collection then re-adding entities that carry their old ids makes Hibernate mix orphan-deletes and re-inserts of the same PK (typically "identifier altered" / constraint errors) once clients send operation ids back. Fix: merge by id (update existing, add new, remove missing) — the logic already drafted in the legacy CashOperationDataAccessMapper.updateEntityFromDomain is the right idea, finish it on the new model.
I-10MinorDuplicate/legacy code: two mapper generations (mapper vs mapperTemp1 — "Temp1" must not ship), CashDemandDTO (unused, wrong package dto.enums), dead enums copied from the identity service, unused ConfigHash, self-assignment bug domain.setEstablishment(domain.getEstablishment()) in DenominationDataAccessMapper, commented-out blocks everywhere.
I-11MinorStyle & consistency: field @Autowired mixed with @RequiredArgsConstructor (pick constructor injection everywhere — also makes tests cleaner), PascalCase method names (CancelRequest, CancelOperations), typos (retreaving, cupure inconu, column total_amount_by_curerency), JUnit 3 (junit:junit:3.8.1) and junit.framework.Assert still on the classpath, enum OperationType missing COLLECTION while the old DDL allowed it (confirm intended values), DTOs exposing bidirectional back-references (CashOperationDTO.cashRequest) → JSON serialization cycles if ever populated.
I-12MinorDuplicate timestamps: CashOperationEntity has both inherited creationDate/modificationDate (Hibernate-managed) and manual createdAt/updatedAt defaulted with LocalDateTime.now(). Keep one pair.
11 · Improve

What's missing entirely

  • Security — no Spring Security, no authentication/authorization. Anyone reaching port 8300 can create or cancel cash movements for any branch. Given UserProfileEntity exists, wiring OAuth2/JWT (uid from the token instead of the payload) is the obvious next step.
  • The validation workflow itself — statuses imply an approval chain (INITIATED → VALIDATED) but there is no endpoint, no actor model, no history of who validated what. This is where Camunda 7 fits: a BPMN process Cash Request Approval (branch manager task → cash-center task → status callback into this service), keeping this microservice as the system of record and Camunda as the orchestrator.
  • Database migrations — no Flyway/Liquibase (root cause of I-03).
  • API documentation — no springdoc-openapi; consumers (the Angular front) have no contract to code against.
  • Observability — no actuator, no health/readiness probes, no metrics, no structured logging/correlation ids.
  • Pagination & filtering — getAll() returns everything; needs Pageable + filters (status, branch, date range) — JpaSpecificationExecutor is already declared on the repositories (untyped, add the generic) and unused.
  • Reference data endpoints — the front will need GET /denominations, GET /currencies, GET /establishments to build the request form; the repositories exist, the controllers don't.
  • Unique request reference — the domain field uniqueRequestRef exists but is never generated/persisted; a human-readable reference (e.g. CR-2026-000123) is essential for branch/back-office communication.
  • Auditing — who did what, when: status-change history table or Envers.
  • CI/CD & containerization — no Dockerfile, no pipeline, no code-quality gate (the @SuppressWarnings("java:S119") hints Sonar is intended but not enforced).

Improvement roadmap — the order I'd do it in

Sprint 1 — Make it correct (fix before feature)

  1. Fix I-01 (broken getAll) and I-02 (ventilation validation) — with unit tests locking both.
  2. Introduce Flyway: V1__baseline.sql generated from current entities + referential seed migration; switch to ddl-auto: validate (I-03).

  4. Bean Validation on the DTO tree + @Valid in the controller (I-07).
  5. Externalize configuration, add local/dev profiles, remove credentials from git history (I-06).

Sprint 2 — Make it clean

  1. Delete the legacy mapper generation; either keep the hand-written "Temp1" mappers renamed properly, or — since MapStruct is already a managed dependency — migrate the mechanical parts to MapStruct and keep only referential-resolution logic by hand (I-04, I-10).
  2. Replace String statuses with enums (RequestStatus, OperationStatus) + a small transition table in the domain; same for isUrgent → boolean.
  3. Constructor injection everywhere, remove field @Autowired; purge dead code and jakarta imports from domain-core with an enforcer rule (I-08, I-11).
  4. Proper merge-by-id update strategy (I-09), single timestamp pair (I-12).
  5. Test pyramid from §9: domain, use case, mapper round-trips, Testcontainers slice.

Sprint 3 — Make it complete

  1. Reference-data endpoints (denominations, currencies, establishments) for the Angular form.
  2. Pagination/filtering on the list endpoint via Specifications; generate uniqueRequestRef (sequence-backed).
  3. springdoc-openapi + published contract; actuator + health probes; Dockerfile + compose (app + postgres) + CI pipeline with tests and Sonar.
  4. Spring Security resource server (JWT); derive the requesting user from the token.

Sprint 4 — Make it a workflow (Camunda 7)

  1. Model Cash Request Approval in BPMN: start on request creation → user task "Validate request" (branch manager) → user task "Approve & schedule transport" (cash center) → service task calling back PATCH /cash/{id}/validate (new endpoint) → end / rejection branch that calls the existing cancel.
  2. Run Camunda 7 either embedded in this service (spring-boot starter, same DB or dedicated schema) or as the platform's central engine with external task workers — given this is a microservice platform, external tasks keep the coupling lowest.
  3. Add a VALIDATED transition + validation history; emit domain events (request created / validated / cancelled) so downstream services (transport, treasury) can react — the empty QlickeerEventName enum suggests an event bus already exists in the platform.
